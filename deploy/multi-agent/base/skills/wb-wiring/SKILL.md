# wb-wiring — схемы подключения Wiren Board кодом

Специализация скилла `diagrams` под одну задачу: **монтажные схемы подключения WB**
(controller ↔ устройства по RS-485, питание, реле, входы). Даёт готовую библиотеку
блоков с **реальными именами клемм** и **проверку результата на типовые ошибки**.

Схемы рисуем **кодом**, не `image_gen` (диффузия рисует «похоже на схему» —
нечитаемо и неверно). Готовые блоки живут в `${HERMES_SKILL_DIR}/references/wb_blocks.py`.

**Один источник — нетлист. Из него И рисуем, И валидируем.** Рисовать трассы
«из головы» отдельно от модели НЕЛЬЗЯ: тогда ERC проверяет один граф, а нарисован
другой, они разъезжаются, и зелёный ERC ничего не гарантирует про картинку (так
уже уезжали закороты и наложения «с тремя галочками»). Поэтому:

1. **Опиши схему как электрическую модель** — кубики + цепи в `wb_netlist.py` (SKiDL).
2. **`nl.validate()`** гоняет **ERC**: `[]` = исправна; строка = реальная ошибка
   (оборванный пин, непитание, закорот шины, нагрузка без обратки).
3. **`ws.render()`** (`wb_schematic`) рисует картинку **ИЗ той же цепи** — 2D-схема,
   раскладка **по напряжению**: HV сверху / LV снизу, модули посередине, обозначения
   `U1..`. Наложений трасс нет **по построению**, пересечения — только под 90°.
4. **Геометрические аудиты** картинки обязаны быть пустыми (см. ниже).

## Quick Start

```python
import sys; sys.path.insert(0, "${HERMES_SKILL_DIR}/references")
import wb_netlist as nl, wb_schematic as ws, wb_blocks as wb

nl.new_circuit()                                   # 1. модель
nl.scenario_relay_load()                           #    (готовый сценарий или собери свой)
# nl.bus(nl.wb("WB7"), nl.wb("WB-MDM3"), nl.wb("WB-MCM8"), add_lan=True)  # свой набор + Ethernet

issues = nl.validate()                             # 2. ERC
assert not issues, f"электрическая ошибка: {issues}"

d = ws.render()                                    # 3. рисуем ИЗ той же цепи (2D по напряжению)
with d:
    d.save("/tmp/wb.png", dpi=wb.STYLE["dpi"])

assert wb.audit_drawing(d) == [], "обрыв линий"    # 4. геометрия
assert wb.audit_overlaps(d) == [], "наложение трасс"     # трассы не лежат друг на друге
assert wb.audit_labels(d) == [], "битая подпись"         # нет литеральных \\n и мусора
assert wb.audit_dots(d) == [], "точка не на узле"        # точка = только эл. соединение
```
Затем в ответе поставь строку `MEDIA:/tmp/wb.png` — гейтвей пришлёт картинку в чат.

## Hard rules — так рисуем всегда (иначе схема неверная)

- **Рисуем ТОЛЬКО из нетлиста. Единственный путь:** собери модель (`nl.wb(...)`/кубики + цепи `nl.Net(...) += ...`) → `nl.validate()` (ERC) → **`ws.render()`**. **НИКОГДА не пиши schemdraw-код руками и не копируй старые картинки-примеры.** Признаки того, что ты рисуешь руками (так НЕЛЬЗЯ): литеральный `\n` в подписи (`WB-MDM3\nDimmer`), выдуманное «2ch» у трёхканального, коробки вместо УГО лампы/вентилятора, нагрузки, приклеенные к блоку. Рисуешь руками → рисунок разъедется с ERC.
- **Трассы не накладываются.** Две трассы не лежат друг на друге; пересечения — только под прямым углом, в одной точке. `wb.audit_overlaps(d) == []` обязателен (рендер из нетлиста гарантирует по построению).
- **Точка (Dot) = только электрическое соединение** — 3+ луча ОДНОЙ цепи (T/+). На углах точек нет; пересечение без точки = просто пересечение. `wb.audit_dots(d) == []` обязателен.
- **RS-485 = 3 провода A/B/GND** (GND общий, опорный для A/B). **Питание V+/GND — отдельно** от шины данных; контроллер получает свой +24В от БП (`nl.bus` делает это сам). Всё это раскладывает `ws.render()` — вручную не разводи.
- **Раскладка по напряжению:** высоковольтное (сеть L/N/PE, лампы, выходы диммера, реле-контакты, моторы) — сверху; низковольтное (24В, RS-485, входы/выключатели, 0-10В, сенсоры, БП) — снизу; модули посередине. Это делает рендер.
- **Терминаторы 120 Ω** — только на длинной/быстрой шине (WB >100 м), на обоих концах; в щите обычно не нужны.
- **Входы — на `iGND`**, не на V+. Реле WB — сухой NO-контакт. **N всегда; PE — при заземлении; фазы** через `mains(phases=1|3, pe=)`.
- **RS-485 — физическая шина, а НЕ протокол.** Modbus 9600 8N2 — частый дефолт WB, но не единственный (приводы штор, `8N1`, кастом — норма); формат бери из datasheet, **не форси Modbus**.
- **Не выдумывай** клеммы/модели/адреса; число каналов фиксирует `nl.CATALOG`; НЕ пиши в схему IP/хосты (см. `references/wb-devices.md`).

## Сценарии (собираются в НЕТЛИСТЕ, рисуются `ws.render()`)

```python
# готовые типовые (каждый валидируется в []):
nl.scenario_relay_load()          # реле → нагрузка (сеть)
nl.scenario_contactor_motor()     # реле → катушка контактора → 3~ мотор (реле не тянет мотор напрямую)
nl.scenario_dimming_0_10v()       # WB-MAO4 0-10В → LED-драйвер → лента
nl.scenario_inputs(n=3)           # выключатели → входы на iGND

# свой набор на шине (контроллер первым; add_lan=True — Ethernet контроллера в LAN):
nl.new_circuit()
nl.bus(nl.wb("WB7"), nl.wb("WB-MDM3"), nl.wb("WB-MR6C v.2"), nl.wb("WB-MCM8"), add_lan=True)
# затем добавь нагрузки/входы/сеть цепями:  nl.Net("L") += ac["L"], relay["COM1"]  ...
# потом:  issues = nl.validate();  d = ws.render()
```

- **Каталог WB** — `nl.wb("WB-MDM3")` и т.п.; число каналов фиксировано (`nl.CATALOG`),
  `nl.wb("WB-MDM3", channels=2)` → ошибка (нельзя выдумать).
- **Модули/устройства**: `psu`, `mains(phases, pe)`, `rs485_device`, `relay_outputs`,
  `discrete_inputs`, `dimmer_outputs`, `led_pwm`, `analog_out`, `analog_in`, `energy_meter`,
  `water_leak`, `contactor`, `led_driver_0_10v`, `blind_drive(control="hv"/"lv")`, `motor`.
- **Базовые элементы — бери ИМЕННО их, не `load()`** (иначе выйдет безликая коробка вместо УГО):
  `lamp`, `fan`, `pump`, `heater`, `switch`, `valve` (2 клеммы `1`/`2`, рисуются реальным символом).
  `load()` — только generic-заглушка для типа, которого в списке нет.
- **3 фазы / PE** — `mains(phases=3, pe=True)`; **Ethernet** — `bus(add_lan=True)`.
- Устройства/схемы нет — **напиши свой кубик**: `references/writing-cubes.md`.

## Проверка результата — обязательный цикл

Модель не видит своих ошибок «на глаз» — проверяй, потом отправляй.
Полная методика и чек-лист — в `references/checklist.md`.

1. **Электрически (ERC):** `nl.validate()` на цепи — `[]` = исправна. Ловит непитание,
   оборванные пины, закорот A/B, нагрузку без обратного провода. Каталог `nl.wb(...)`
   фиксирует число каналов → нельзя выдумать «WB-MDM3 на 2 канала». Свой кубик — `writing-cubes.md`.
2. **Геометрически (на картинке из `ws.render()`):**
   - `wb.audit_drawing(d)` — обрыв/провод-в-никуда;
   - `wb.audit_overlaps(d)` — **наложение трасс** (провода друг на друге; пересечения только под 90°);
   - `wb.audit_dots(d)` — **точки только на узлах** (соединение одной цепи; не на углах/пересечениях);
   - `wb.audit_labels(d)` — литеральный `\n`/мусор в подписях;
   - `wb.validate_devices([...])` / `wb.check_caption(...)` — терминаторы, адреса, утечки IP/хостов.
3. **Глазами:** сохрани PNG, **перечитай его** (открой картинку) — провода приходят на
   клеммы, нет линий в никуда, подписи читаются, всё реальное.

Есть провал → правь код → повтори (2–3 итерации). **Первый рендер не глядя не отправляй.**

## Доставка

Сохраняй в **абсолютный** путь и ставь `MEDIA:/tmp/wb.png` в ответе. PNG — превью в
Telegram; SVG (`d.save(".../wb.svg")`) — как исходник по запросу, Telegram его не превьюит.
Стиль тёплый пастельный (`wb.STYLE`), фон бумажный; подписи латиницей (в деплой-образе
может не быть кириллического шрифта для schemdraw).

## References

- `references/wb_schematic.py` — **единственный рендер** (`render(circuit)`): 2D-схема из нетлиста. Раскладка **по напряжению** — HV сверху (сеть, лампы, фазовые выходы, реле-контакты, моторы), LV снизу (24В, RS-485, входы/выключатели, 0-10В, сенсоры, БП), модули посередине, mixed-приборы страдлят середину. WB-шина — один толстый трунк `V+/GND/A/B` (RS-485 = A/B/GND, общий GND). Ethernet у контроллера через `nl.bus(add_lan=True)`. Обозначения `U1..`. Чисто по построению + разрешение коллизий сдвигом; масштаб до ~15 модулей.
- `references/wb_blocks.py` — только база schemdraw: `STYLE`, `new_drawing()` и **аудиты** `audit_drawing` / `audit_overlaps` / `audit_dots` / `audit_labels` / `validate_devices` / `check_caption`. Никакого ручного рисования блоков/сценариев тут больше нет — единственный рендер `wb_schematic`.
- `references/wb_netlist.py` — электрическая модель (SKiDL) + `validate()` (ERC + WB-правила). Кубики `psu/mains/rs485_device/relay_outputs/discrete_inputs/dimmer_outputs/led_pwm/analog_out/load/motor/contactor/led_driver_0_10v`, сценарии `scenario_*`, каталог `wb()` с фиксированным числом каналов.
- `references/writing-cubes.md` — как написать свой кубик под устройство, которого нет в каталоге (электрические типы пинов, доменные правила).
- `references/checklist.md` — методика проверки + галерея типовых ошибок LLM (что НЕ делать).
- `references/wb-devices.md` — проверенные факты по клеммам/моделям + **полный каталог WB Modbus-устройств** (со ссылками на wiki).

## Что НЕ делаем

- Не рисуем схемы через `image_gen` / растровую генерацию.
- **Не тянем трассы руками отдельно от нетлиста** — только `ws.render()` из модели (иначе ERC врёт).
- **Не накладываем трассы друг на друга**; пересечения — только под прямым углом.
- Не рисуем шину одной линией и устройства голыми прямоугольниками.
- Не выдумываем клеммы/адреса/модели; не пишем в схему реальные IP/хосты.
- Не отправляем результат без `audit_drawing` + `audit_overlaps` + `audit_labels` и визуальной проверки.

## Когда спросить пользователя

- Неизвестны модель устройства, адреса, число фаз или наличие заземления — уточни, не угадывай.
- Устройств больше ~6 / схема нечитаема — предложи разбить на несколько картинок или дать таблицу.
- Клеммы конкретной модели не подтверждаются `wb-devices.md`/datasheet — попроси datasheet или пометь «пример».
