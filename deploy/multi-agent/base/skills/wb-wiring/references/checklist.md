# Проверка схемы на ошибки — методика

LLM «чинит» схему и не видит, что осталось сломано (реальные примеры — в конце).
Поэтому результат проверяется в два уровня, и только потом уходит в чат.

## Цикл

```
собрать → audit_drawing() + validate_devices() → сохранить PNG →
ПЕРЕЧИТАТЬ картинку → пройти визуальный чек-лист →
есть провал? → править код → повтор (2–3 итерации) → MEDIA: в чат
```
Первый рендер не глядя не отправляем.

## Уровень 1 — программно (детерминированно)

```python
floating = wb.audit_drawing(d)          # висящие концы / обрыв / провод-в-никуда
assert not floating, floating

problems = wb.validate_devices([         # инварианты спецификации
    dict(name="ctrl",  address=1, label="ttyRS485-1"),
    dict(name="relay", address=5, label="WB-MR6C v.2"),
], terminators=2)
assert not problems, problems
```

`audit_drawing` — главный: каждый конец провода обязан лежать на клемме, точке-узле
или на другом проводе; всё прочее — висящий конец. `validate_devices` проверяет:
ровно 2 терминатора, у каждого устройства подключено питание, адреса уникальны,
в подписях нет IP/внутренних хостов.

## Уровень 2 — визуальный чек-лист (открой PNG и пройди по пунктам)

- [ ] Каждый провод **приходит на клемму** и останавливается — не пересекает корпус насквозь.
- [ ] Нет линий, уходящих **в никуда**; точка стоит только на реальном узле (T-соединение).
- [ ] Рамки устройств и провода **различимы** (разный цвет; корпус — не цвет провода).
- [ ] Подписи не наезжают на провода/друг на друга и не вылезают за блок (шире блок / короче текст).
- [ ] Полярность видна: `A→A`, `B→B`, `GND→GND` по всей цепочке.
- [ ] Терминаторы: на короткой шине в щите их **НЕТ**; на длинной/быстрой (>100 м) — на **обоих** концах. Питание — отдельные рельсы; GND общий.
- [ ] `N` присутствует; `PE` — если есть заземление; фазы соответствуют (1~ / 3~).
- [ ] Все подписи (модель, адрес, клеммы, `9600 8N2`) **реальные** — сверены с datasheet/конфигом, ничего не выдумано, нет IP/хостов.

## Галерея типовых ошибок LLM (так НЕ делать)

Собрано по реальным попыткам нарисовать RS-485 разными моделями:

1. **Шина одной толстой линией** с подписью «A/B/GND». Не видно `A→A/B→B`; перепутанные
   A/B (главная ошибка монтажа) не показать и не проверить. → три отдельных проводника.
2. **Провода сквозь корпус** устройства. Неоднозначно: подключено или нарисовано поверх?
   → линия останавливается на клемме.
3. **Голые прямоугольники без клемм** / прижатые общими стенками. → блок с именованными пинами.
4. **Висящие отводы и точки в пустоте** (провод в воздух). → каждый конец на клемме/узле.
5. **Слипшиеся подписи пинов** («3AGNDINEMBK3») — блок мал под число клемм. → шире блок / короче имена.
6. **Ничего не соединено** — «россыпь деталей» вместо схемы. → сначала топология, потом рендер.
7. **Нет питания вообще.** → отдельные рельсы V+/GND. (Терминаторы 120 Ω на коротких шинах в щите НЕ нужны — только для длинных/быстрых линий >100 м; их отсутствие там не ошибка.)
8. **Цвет корпуса == цвет провода** (путаница). → корпус нейтрального цвета.
9. **Выдуманные значения**, которых у тебя нет (модели/адреса/клеммы). → бери из datasheet/конфига.
   НО: RS-485 — физическая шина, **не** протокол. Не-Modbus устройства (привод шторы,
   кастомный протокол) и форматы вроде `8N1` — норма, **не ошибка**; не форси Modbus 8N2.
10. **Утечка инфраструктуры** в подпись (`IP:192.168.2.106`). → в схеме только логический
    идентификатор порта/шины, никаких IP/хостов (ловится `validate_devices`).
