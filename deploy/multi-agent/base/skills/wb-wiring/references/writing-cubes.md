# Писать свои кубики для ERC-модели

`wb_netlist.py` — это *электрическая модель* схемы на SKiDL. schemdraw рисует
картинку, а эта модель проверяется ERC (Electrical Rules Check): собрал цепь из
кубиков и цепей, вызвал `validate()` — исправная схема даёт `[]`, любая строка в
ответе = реальная ошибка (оборванный пин, непитаемое устройство, закорот шины,
нагрузка без обратного провода).

Готовые кубики покрывают типовое: `psu`, `mains`, `rs485_device`,
`relay_outputs`, `discrete_inputs`, `dimmer_outputs`, `led_pwm`, `analog_out`,
`load`, `motor`, `contactor`, `led_driver_0_10v`. Готовые сценарии (собираются и
валидируются в `[]`): `scenario_relay_load`, `scenario_contactor_motor`,
`scenario_dimming_0_10v`, `scenario_inputs`. Когда устройства/схемы нет —
**напиши свой кубик**, это несколько строк. Механизм универсальный, не привязан к WB.

## Кубик = имя + пины с электрическим типом

```python
import wb_netlist as nl
from skidl import Pin
F = Pin.funcs

def my_sensor(name="MY-SENSOR"):
    # V+ GND A B — стандартная база RS-485; сверху свои клеммы
    return nl.rs485_device(name, extra=[
        ("AO", F.OUTPUT),      # аналоговый выход
        ("AGND", F.PWRIN),     # его земля
    ])
```

Совсем с нуля (не RS-485-устройство), через `nl._cube` — тот же приём:

```python
def contactor(name="KM1"):
    return nl._cube(name, [
        ("A1", F.PWRIN), ("A2", F.PWRIN),   # катушка
        ("L1", F.PASSIVE), ("T1", F.PASSIVE),  # силовой контакт
    ])
```

## Электрические типы пинов (главное, что нужно знать)

От типа зависит, что ERC считает ошибкой:

| Тип | Когда |
|---|---|
| `PWRIN` | вход питания — **должен** получить питание, иначе «unconnected/undriven» |
| `PWROUT` | выход питания (источник): `psu` V+/GND, `mains` L/N, диммер O* |
| `OUTPUT` | сигнальный выход; два `OUTPUT` на одной цепи = конфликт |
| `INPUT` | сигнальный вход |
| `BIDIR` | двунаправленный — так помечены A/B шины |
| `PASSIVE` | нейтральный контакт: клеммы реле, ножки нагрузки |

Правило: **источник — `PWROUT`, потребитель — `PWRIN`, контакт/нагрузка —
`PASSIVE`.** Тогда неподключённое питание всплывает само.

## Чтобы чистая схема давала ровно `[]`

- **Каждой силовой цепи нужен драйвер.** Рейки питания и шину помечай
  `net.drive = nl.POWER` (или бери готовые `power_rails()` / `rs485_pair()`), иначе
  ERC ругнётся «No drivers for net».
- **Каждая цепь ≥ 2 пинов.** Один пин на цепи = провод в никуда (это фича — так
  ловится оборванная нагрузка).
- **Неиспользуемые клеммы — глуши явно:** `nl.unused(dev["K3"], dev["K4"])`.
  Хелпер `bus()` делает это сам для запасных каналов на магистрали.

## Правила, которые ERC знать не может — в `_wb_domain_issues`

ERC — общий: он считает две `BIDIR`-ноги на одной цепи законными, а для RS-485
`A`+`B` на одном узле — это ошибка монтажа. Такие доменные правила добавляй в
`_wb_domain_issues()` обычным Python по объектам цепи (`_circuit().parts`,
`pin.net`) — **не** строковым `erc_assert` (он хрупкий, eval не видит имён).

```python
# пример: у диммера L и N не должны сойтись на одну цепь
for part in _circuit().parts:
    bn = {p.name: p for p in part.pins}
    if "L" in bn and "N" in bn and bn["L"].net is not None \
            and bn["L"].net is bn["N"].net:
        issues.append(f"L/N shorted on {part.ref}")
```

## Каталог WB и запрет выдумок

`CATALOG` фиксирует число каналов модели. `wb("WB-MDM3", channels=2)` кинет
ошибку — MDM3 трёхканальный. Добавляешь модель — бери факты из
`wb-devices.md`/datasheet, **не угадывай число каналов**.

## Обязательный цикл

```python
nl.new_circuit()
... собери схему ...
issues = nl.validate()
assert not issues, issues        # [] == электрически исправна
```

Прогони это **до** отправки картинки. schemdraw-рендер и ERC-модель — про одну и
ту же схему; расходятся — значит где-то ошибка в одной из них.
