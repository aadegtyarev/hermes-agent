# Заметка: инструмент рисования электрических схем для LLM

Ретроспектива по разработке `wb-wiring` — как довести LLM до **проверяемо
корректных** монтажных схем WB, а не «похоже на схему». Проблемы, подходы,
принятые решения, тесты, сложность. Пишется по ходу — часть решено, часть открыто.

## 1. Задача и почему она нетривиальна для LLM

Нужно: по запросу «как подключить X к Wiren Board» отдать в чат картинку-схему.
Масштаб реальный — ≤ 10–15 модулей (щит, а не ASIC), должно оставаться читаемым.
Потребитель — бот техподдержки.

Почему сложно именно для LLM:
- **Диффузия (`image_gen`) не годится** — рисует «похоже на схему»: нечитаемо и
  электрически неверно.
- **Модель не видит своих ошибок «на глаз»** — уверенно отдаёт сломанную картинку.
- Рисование кодом (schemdraw) убирает диффузию, но переносит проблему: модель
  теперь генерит **координаты линий**, и ошибается в них так же уверенно.

## 2. Ключевой провал: рисование «из головы» → дрейф

Первая версия: модель рисовала schemdraw'ом вручную, а корректность проверяли
отдельно. Итог из поля: схема WB-MDM3 с литеральным `\n` в подписи, выдуманным
«2ch» (диммер трёхканальный), закороченными выходами и **наложением трасс** — и
всё это «с тремя зелёными галочками».

Урок №1: **если картинку рисуют отдельно от того, что проверяют, — они разъедутся,
и зелёная проверка ничего не гарантирует про картинку.** Нужен один источник.

## 3. Архитектура: один источник — нетлист

```
описать схему как НЕТЛИСТ (SKiDL: кубики + цепи)
        │
        ├─ nl.validate()  → ERC (electrical rules check): непитание, обрыв,
        │                     закорот шины, нагрузка без обратки, A/B на одной цепи
        │
        └─ render(circuit) → рисуем ИЗ ТОЙ ЖЕ цепи (schemdraw)
                                │
                                └─ audits(картинка) → геометрические инварианты
```

- **SKiDL** — модель + ERC. Выбран как зрелый готовый ERC-движок (MIT, pip);
  свои устройства описываются без KiCad-библиотек, пины имеют электрический тип.
- **schemdraw** — только отрисовка в нашем стиле. Рендер SKiDL (`netlistsvg`)
  отвергнут — тянет Node-`netlistsvg`, и раскладка не наша.
- Картинка и проверяемый граф — **один объект**, дрейфа нет.

## 4. Правила-инварианты (заветы)

Зафиксированы как жёсткие, проверяются машинно:
1. **Рисуем только из нетлиста**, не тянем трассы руками.
2. **Трассы не накладываются** — две трассы не лежат коллинеарно друг на друге;
   провода только пересекаются, и только под прямым углом.
3. **Точка (Dot) = только электрическое соединение** — 3+ луча одной цепи (T или +).
   На углах точек нет; пересечение без точки — просто пересечение.
4. **Питание (V+/GND) отдельно от RS-485 (A/B)** — разные физические трассы на WB.
5. **Не выдумывать** клеммы/каналы/модели — брать из каталога/datasheet
   (каталог `wb()` фиксирует число каналов: `wb("WB-MDM3", channels=2)` → ошибка).
6. **Позиционные обозначения** (U1, U2, G1…) — при >1 компонента, для однозначности.
7. Никаких IP/хостов в подписи.

## 5. Аудиты как гарантии + тесты

Модель не проверяет себя «глазами» → инварианты проверяются кодом (`wb_blocks`):
- `audit_drawing` — висящие концы / провод-в-никуда;
- `audit_overlaps` — наложение коллинеарных трасс (H и V) + диагонали;
- `audit_dots` — точки только на узлах одной цепи (не на углах/пересечениях);
- `audit_labels` — литеральный `\n`/escape/control в подписях;
- `validate_devices` / `check_caption` — терминаторы, адреса, утечки инфраструктуры.

**Тесты — двух видов, оба обязательны:**
- позитивные: каждый эталонный сценарий, отрисованный из нетлиста, проходит ВСЕ аудиты;
- негативные: каждый детектор реально **срабатывает** на засеянной ошибке
  (наложение H/V, литеральный `\n`, дот на углу, дот на пересечении, закорот A/B).
  Детектор, который никогда не флагует, — бесполезен; это ловится негативным тестом.
- образный: `tests/docker/` собирает зависимости в контейнере и гоняет оба слоя
  внутри реального образа (ловит «добавил зависимость, а она не ставится/не грузится»).

## 6. Эволюция рендера

1. **Lane-рендер** (`wb_render`, УДАЛЁН): первый подход — каждая цепь своя
   горизонтальная дорожка, каждый вывод своя вертикаль; чисто по построению, но
   «backplane» в один ряд без шины. Его идея (уникальная дорожка/колонка) вошла в
   `wb_schematic`; сам модуль убран, чтобы не было двух путей рисования.
2. **Шина**: сворачивать V+/GND/A/B в трунк с отводами, чтобы не тащить 4
   параллельных провода. Упёрлись в то, что шина — мульти-проводниковая
   абстракция, которую одножильные аудиты не понимают; классический 45°-ripper
   нарушает «только 90°». Решение: трунк = одна линия + прямоугольные отводы,
   связность гарантирует ERC, а не картинка.
3. **2D-раскладка** (`wb_schematic`, экспериментальный): источники сверху, модули
   в ряд посередине, БП снизу, питание/шина в нижнем банке, нагрузки/рейки сверху.
   Позиционные обозначения. Ближе к «настоящей» схеме.

## 7. Ядро сложности: размещение ↔ разводка связаны

Простая гарантия «без наложений» из lane-рендера держится, только пока
**стороны пинов жёстко привязаны** к банкам (хардкод «power+bus → вниз, остальное
→ вверх»). Как только стороны выбираются **анализом топологии** (сторона пина ←
куда смотрит его цепь), цепь может идти и вверх и вниз, и банковый роутер
перестаёт гарантировать уникальность колонок: два стояка из разных рядов попадают
на почти один x → наложение.

Это не баг реализации, а суть: **красивая раскладка = размещение + ортогональная
разводка с обходом, а минимизация пересечений — NP-полная.** Ровно поэтому
существует ELK (годы затачивания). Дёшево гарантировать чистоту можно, только
ограничив одну из степеней свободы (фиксированные стороны/ряд) — ценой гибкости.

## 8. Оценка готовых решений (почему свой)

| Инструмент | Даёт | Почему не взяли |
|---|---|---|
| **ELK / netlistsvg** | авторазмещение + ортогон. разводка + шины | Node-интероп; на спайке **сам дал 2 наложения** (бандлит); оверкилл на ≤20 узлов |
| **graphviz `ortho`** | ортогональные рёбра, уже в образе | не держит порты/подписи рёбер — для схем слабо |
| **force-directed / d3** | быстрая раскладка узлов | «органика» с диагоналями — противоречит «только 90°»; это про графы, не про монтаж |

Решение: движок раскладки **не тащить**. Топология WB структурная (шина + модули
+ локальные нагрузки), а не произвольный граф — под неё детерминированная
раскладка проще и предсказуемее, и остаётся в наших правилах.

## 9. Статус (РЕШЕНО) — 2D-рендер чист и в проде

Открытая проблема «наложения при 2D-размещении» закрыта прагматично, без ELK:
- **`wb_schematic` (2D) — в проде, рекомендован.** Источники сверху, модули в ряд
  (обозначения U1..), БП снизу, питание и RS-485 раздельными нижними банками.
- **Как добились чистоты:** каждая цепь — уникальная дорожка y (горизонтали не
  совпадают); стояки падают на x пина; единственная возможная коллизия —
  пин верхнего/нижнего ряда попал на x модульного пина — **разрешается сдвигом
  этих рядов на малую фазу**, пока `audit_overlaps()==[] и audit_dots()==[]`
  (коллизии редки, сходится за единицы попыток). Плюс дот ставится только при
  ненулевом стояке и отступе от концов дорожки ≥ eps аудита (иначе тап у края
  читается как угол).
- Проверено: все сценарии + наборы «3 диммера», «контроллер+2 реле+MCM8»,
  7-модульный микс — все аудиты пусты (`tests/skills/test_wb_render_skill.py`).
- **Один путь рисования** — только `wb_schematic.render()`. Запасной lane-рендер
  `wb_render` удалён: два рендера сбивали агента «какой брать».
- Урок: для ≤15 узлов «структурная раскладка + разрешение редких коллизий сдвигом»
  дешевле и предсказуемее общего автороутера/ELK, и остаётся в правилах.

## 10. Сложность / трудозатраты (оценка)

- Декларативная модель + ERC + аудиты + lane-рендер + каталог: **сделано**, надёжно.
- Шина (трунк + отводы): дни.
- 2D-раскладка «как ELK» с гарантией чистоты: от рабочего варианта — дни; до
  качества ELK — месяцы (что подтверждает: свой полноценный автороутер писать
  дорого, брать ELK — тоже дорого по интеграции; для ≤20 узлов дешевле
  детерминированная структурная раскладка + честные аудиты).

## 11. Слепая зона аудитов: подпись↔подпись (визуальный просмотр обязателен)

Аудиты §5 проверяют **связность и геометрию линий** (висящие концы, наложения,
доты, литеральный `\n`). Они НЕ проверяют, что **две подписи не налезли друг на
друга** — текст для них не объект. Это отдельный класс дефектов, и он всплывал уже
дважды, оба раза найден **только глазами**, при зелёных аудитах:

- **#60** — тег трунк-шины «24V + RS-485 (A/B/GND)» налезал на подписи-обрубки
  отводов. Починка: тег под трунк, лишние обрубки убрать.
- **#61** — заголовок модуля по центру коробки налезал на повёрнутые на 90°
  подписи верхних пинов, когда те длинные: `U4 · WB-MR6C v.2` поверх `COM1`/`K2` на
  реле. У диммера/PSU/сети такого нет — там верхние подписи короткие (`L/N/O1`,
  `V+c`) и до центра не достают. Починка (`_draw_part`): `long_top = любой верхний
  пин с именем ≥4 символов` → заголовок над коробкой, иначе по центру.

Отсюда два вывода:
1. **Визуальный просмотр рендера — не опция, а обязательный слой проверки** наравне
   с ERC и аудитами. «Аудиты зелёные» ≠ «схема годная». Прогонять разнотипные сцены
   (реле с длинными пинами, 3-фазка, диммеры, входы, счётчики) и **смотреть картинку**.
2. **`matplotlib get_window_extent` для повёрнутого текста возвращает bbox ДО
   поворота** — авто-проба «пересекаются ли рамки подписей» на этом промахивается
   (для #61 показала «нет пересечения», хотя на картинке налипание очевидно). Так что
   и тут: доверяй рендеру, а не bbox-пробе. Автоматизировать этот аудит можно (растровая
   проверка — рисуем каждую подпись в свой слой и ищем пересечение непрозрачных
   пикселей), но пока не сделано — держим ручной визуальный контроль.

Почему не добавили «audit_label_overlap» сразу: надёжно это только растрово (по
пикселям отрендеренного текста, с учётом поворота и фактического шрифта), а не по
эвристике длины строки. TODO, если класс всплывёт третий раз.

## 12. Догоняющая сводка раскладки (#47–#61)

Доки §6/§9 писались на этапе lane→2D; актуальное состояние с тех пор:
- **Раскладка по НАПРЯЖЕНИЮ, а не по роли.** Высоковольтные полевые устройства
  (сеть L/N/PE, лампы, фазовый выход диммера, контакты реле, контакторы, моторы) —
  СВЕРХУ; низковольтные (24V, RS-485, сухие входы/выключатели, 0-10 В, DC-LED,
  датчики, БП) — СНИЗУ; модули посередине. Так входы/выключатели идут ПОД модулями.
  Смешанное устройство (например 230 В LED-драйвер с управлением 0-10 В) сидит в
  среднем банке и «раскоряками» тянет ВВ-пины вверх, НВ-пины вниз.
- **RS-485 — три провода A/B/GND** (GND — общая опорная земля), поэтому WB-шина —
  ОДИН низковольтный трунк V+/GND/A/B, а не два раздельных. 2-проводной A/B —
  осознанное исключение для устройств с локальным питанием 230 В.
- **Контроллер питается отдельным +24 В** (`psu(ctrl_pair=True)` даёт клемму `V+c`,
  цепь `V+_ctrl`), НО его GND+A+B остаются на общей шине — RS-485 нужен общий GND;
  изолировать землю контроллера НЕЛЬЗЯ.
- **Реальные УГО** вместо коробок: `lamp/fan/pump/heater/switch/valve` рисуются
  символами schemdraw (тег `_symbol`, 2 вывода «1»/«2»); `load` — generic-коробка.
  В SKILL прямо велено брать типовой куб, а не `load()`, иначе снова коробка.
- **Локальность + читаемость:** полевые устройства ставятся у своего модуля
  (короткие трассы); подпись цепи крепится к углу дорожки (не отдельной колонкой
  слева); безымянные цепи (`N$7`) показываются по характерному выводу (`O1/K1`);
  высота ряда адаптивна (ряд выше стопки дорожек своего банка); коллизии стояков
  гасит порядовый + поблочный сдвиг.
- **Один рендер** — `wb_schematic.render()` (алиас `ws`); `wb_render` удалён.
- Тесты: `tests/skills/test_wb_schematic_stress.py` (реалистичные миксы + Ethernet +
  фазз валидных нагруженных сцен). Старый `test_wb_render_skill.py` удалён вместе с
  lane-рендером.

## 13. Уроки для LLM-инструментов рисования

1. **Не давай модели рисовать пиксели/координаты.** Декларативный ввод (нетлист),
   детерминированный рендер, машинная проверка.
2. **Проверяй то, что нарисовано, а не то, что задумано** — иначе дрейф.
3. **Инварианты → аудиты → тесты (позитивные И негативные).** Зелёная галочка
   ценна, только если детектор доказанно ловит нарушение.
4. **Ограничение бьёт гибкость там, где нужна гарантия.** Для читаемости+чистоты
   при малом N структурная раскладка надёжнее общего автороутера.
5. **Честность в статусе:** не деплой то, что нарушает собственные правила, даже
   если «почти работает».
6. **У аудитов есть слепая зона — подпись↔подпись (§11).** Машинные проверки не
   заменяют визуальный просмотр рендера; «аудиты зелёные» ≠ «схема годная». Всегда
   смотри картинку на разнотипных сценах перед деплоем.
