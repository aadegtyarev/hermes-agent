## Авторизация

Логин оператора в YouTrack задаётся в env (`YOUTRACK_LOGIN`), в навыке не хранится. Для «моих задач» используй `assignee='me'` в `yt_search` — плагин сам подставит `YOUTRACK_LOGIN` (потому что токен может быть авторизован под сервисным пользователем, и `for: me` ненадёжен).

## Формат выдачи задач

При запросе «покажи мои задачи» / «что на мне»:

1. Фильтр по статусам: **Open, In Progress, Отложена**. Остальные статусы (Done, Cancelled и т.д.) не показывать без отдельной просьбы.
2. Исключить задачи с типом **«Работа»** (work tracking, не содержательные задачи).
3. Группировка: **Open** → **In Progress** → **Отложена** (именно в таком порядке).
4. Каждая задача: ссылка `[ID](url)` — summary — изм. `DD.MM[ HH:MM]`. **URL бери из поля `url`** в ответе `yt_search`/`yt_get_issue` (плагин отдаёт готовую ссылку из `YOUTRACK_URL`); хост не хардкодить.
   - Дату изменения показывать всегда.
   - Время показывать только если задача менялась сегодня или вчера (свежесть важна), иначе только дату.
5. Без эмодзи.
6. Заголовки статусов — просто словом (Open, In Progress, Отложена), жирным.

## Поиск «моих задач»

Передавай `assignee='me'` (плагин подставит `YOUTRACK_LOGIN`), а состояния — через `query`:

```text
State: Open OR State: {In Progress} OR State: Отложена
```

Комбинацию состояний давай в `query`, а не в `state` (параметр `state` не поддерживает OR).

**Питфол:** голый `for: me` ненадёжен (токен под сервисным пользователем) — именно поэтому `assignee='me'` резолвится в `YOUTRACK_LOGIN` на стороне плагина. Логин в навыке не пишем.

## Поиск идей (Type: Идея)

Когда пользователь спрашивает «сколько идей / какие идеи в проекте»:

1. **Всегда указывай State в запросе.** `project: INT Type: Идея` без State вернёт все 70+ штук за все годы — это портянка.
2. **Правильный запрос:**
   ```text
   project: INT Type: Идея State: Open
   ```
   Это даст ~14 актуальных идей, не десятки закрытых/завершённых.
3. **Сначала назови количество.** Прежде чем выводить список, скажи «X открытых идей». 
4. **Выводи только после подтверждения.** Не сваливать список без спроса. Если пользователь спросил «сколько?» — ответь числом, не списком. Если спросил «какие?» — сначала назови число, потом список.
5. **Формат списка идей — идентичен задачам:** каждая строка: `• [INT-NNNN](url) — summary`. Ссылка встроена в ID, не выносится отдельно. Не группировать URL блоком в конце.

**Питфол:** `state` параметр в `yt_search` не поддерживает пробельные значения (`In Progress`). Использовать `query` с фигурными скобками: `State: {In Progress}`.

## Питфолл: при поиске темы из TG — проверять несколько проектов

Если тема обсуждалась в каком-то чате (PRJ-xxx, WB Dev и т.д.), это **не значит** что тикет лежит в том же проекте YT. Пример: обсуждение AI-ревью в чате **PRJ-xxx**, тикет INT-1234. 

Правило: не нашёл в первом проекте — сразу проверь INT, SOFT, INF. INT — проект команды интеграций (AI/CI/инструментарий). SOFT — софт контроллера и облако. INF — инфраструктура.

После `yt_list_projects()` помни какие проекты есть и какое их содержание. Используй короткие имена: PRJ, INT, SOFT, INF, CLOUD.

## Питфолл: web_extract не работает (Firecrawl без кредитов)

web_extract идёт через Firecrawl (JS-рендеринг). Если кредиты кончились — все запросы падают с `Payment Required`.

**Когда Firecrawl не работает — альтернативы:**
- Статика (GitHub README, вики): `curl -sL <URL>` + grep/sed
- Discourse (support.wirenboard.com): `curl -sL <URL> | grep -oP 'github\.com[^"<> ]+' | sort -u` — ссылки на код есть в HTML
- Чистый текст из HTML: `trafilatura` (Python-библиотека) — без JS, только DOM-парсинг
- web_search — для поиска контекста, если не открывается конкретная страница

Ссылка на код может быть прямо в первом сообщении, но невидима для web_extract.

## Принцип: не сваливать всё без фильтра

- **Никогда** не выполнять запрос без фильтра по State, если явно не сказано «покажи все включая закрытые».
- **Никогда** не сваливать список >15-20 элементов без предварительного озвучивания количества и запроса «выводить?».

## Mandatory pre-flight checklist (перед любым выводом списка YT-задач)

5. **Format:** ссылки встроены в ID (не отдельным блоком)? Каждая задача = `[ID](url) — summary — date`?
6. **Grouping:** сгруппировано по статусу (Open → In Progress → Отложена)?

## Проверка типа задачи

После получения списка задач, проверь поле `type` через `yt_get_issue` у каждой. Если `type = "Работа"` — это task-tracking (учёт времени на планёрки/встречи), а не содержательная задача. Исключать из выдачи.

```python
issue = yt_get_issue('INT-1234')
# issue['customFields'] → ищем поле name="Type", у него value.name
# Если value.name == "Работа" — исключаем
```

### Оптимизация

Если запрос вернул много задач (>5), можно НЕ дёргать yt_get_issue на каждую для проверки типа — сначала покажи, пользователь скажет если нужно отфильтровать.

## Формат вывода

1. **Группировка по статусу:** Open → In Progress → Отложена (именно в таком порядке).
2. **Заголовок статуса** — просто словом (Open / In Progress / Отложена), жирным. Без эмодзи.
3. **Каждая задача:** `• [ID](https://<youtrack-host>/issue/ID) — summary — изм. DD.MM[ HH:MM]` — ссылка встроена в ID.
4. **Дата изменения** — всегда.
5. **Время** — ТОЛЬКО если задача менялась сегодня или вчера (свежесть важна для приоритизации).
6. **Не группировать ссылки отдельно.** Каждая задача тащит ссылку в своём ID — это и есть навигация. Отдельный блок URL не нужен, он не читается.
7. Пример:
```
Open
• [INT-1234](https://<youtrack-host>/issue/INT-1234) — <summary> — изм. 01.06
• [INT-1235](https://<youtrack-host>/issue/INT-1235) — <summary> — изм. 10.06

Отложена
• [INT-1236](https://<youtrack-host>/issue/INT-1236) — <summary> — изм. 10.06 15:41
```

## Создание задачи с тегом

`yt_create_issue(summary='...', project='INT')` — создаёт задачу и автоматически вешает тег `ai_auto`.

**Параметры:**
- `summary` (обязательно) — заголовок задачи
- `project` (опционально, по умолч. `INT`) — короткое имя проекта
- `description` (опционально) — описание
- `tag` (опционально, по умолч. `ai_auto`) — какой тег применить
- `type` (опционально) — тип задачи. Если **не указан** — применяется дефолт проекта (настроен в YouTrack). Доступные значения: `Идея`, `Bug` (ошибка), `Task` (задание).

**Поведение:**
- **Исполнитель:** создаётся **без исполнителя** (поле `Assignee` — `MultiUserIssueCustomField`, ставится пустой список `[]`), чтобы авто-тикет не садился на владельца токена.
- **Тип:** задаётся только если передан `type` (иначе — дефолт проекта). Три значения: `Идея`, `Bug` (ошибка), `Task` (задание) — точные имена из бандла «Типы» (смесь англ./рус.). Неверное имя → 400, тула вернёт подсказку со списком.
- Тег ищется по имени через `GET /api/tags?query=ai_auto`. Если не найден — создаётся через `POST /api/tags`.
- Тег применяется через `POST /api/issues/{id}/tags` с телом `{"id": tag_id}`.
- Возвращает `issue_id`, `url`, применённый `tag` и `type`.

**Питфолл — формат тела:** тег применяется как `{"id": "7-680"}`, не `{"tag": {"id": "7-680"}}`. Второй вариант падает с 400 Bad Request "укажите ID".

**Питфолл — поиск тега:** Не листать все теги (`$top` не гарантирует что вернёт последние). Использовать `?query=имя_тега`.

**Питфолл — INT дефолт:** Все автоматические тикеты — в проект INT. Если пользователь не указал project — `INT`.

## Рецензия существующего тикета

Когда пользователь просит «прочитай, всё ли ок / логично ли» по существующему тикету — порядок:

1. `yt_get_issue(issue_id)` — полное описание
2. Читать сверху вниз:
   - **Структура:** логичный порядок (суть → проблема → охват → выгода → конкуренты → техрешение)?
   - **Опечатки:** `пользвателей` → `пользователей`, битая латиница в ссылках (`Рisyoch-Проброс`)
   - **Формулировки:** `не tcp-сервисам` — MQTT и SSH это TCP. Автор имел в виду «не-HTTP» или «не только веб-интерфейсы»
   - **Цифры/оценки:** охват 10-20% может быть завышен — укажи если есть другие данные, но не настаивай (это эпик на стадии идеи)
   - **Security:** если не упомянут — отметь как пробел для реализации, не для эпика
3. **Alignment-проверка (соответствие исходному обсуждению):**
   - Проверь секцию «Ссылка на обсуждение» в описании тикета — там могут быть ссылки на Telegram-сообщения
   - Из ссылок `t.me/c/<chat_id>/<msg_id>` достань `chat_id` и `msg_id`
   - `telegram_thread(chat_id=..., message_id=<msg_id>)` — восстанови тред (или `telegram_recent`/`telegram_search`). Доступны только уже проиндексированные сообщения.
   - **Сравни по пунктам:** что говорил автор идеи в TG vs что написано в тикете. Обращай внимание на:
     — **Модель/инструмент:** автор мог настаивать на конкретной (Claude Opus /code-review), а тикет оставил выбор открытым
     — **Место запуска:** автор мог быть против CI (только локально/на стороне отправляющего), а тикет оставил CI как опцию
     — **Пропуск человека:** автор согласен на автоматический merge для AI-слопа или нет
     — **Градации:** автор упоминал градации качества ревью в зависимости от репозитория
   - **Формат вывода:** таблица сравнения «Пункт автора идеи → В тикете → Совпадает?» (✅/⚠️/❌) + итог
4. **Формат ответа:** сверху вниз без пропусков:
   - Сначала общая оценка (логично/есть проблемы)
   - Потом конкретные замечания с цитатами из описания
   - Потом alignment-сравнение с исходным обсуждением (если применимо)
   - В конце резюме
5. Вывод: годное / нужно править / переписать

## Питфолл: установка стороннего софта на Wiren Board

Когда описываешь процедуру установки стороннего ПО на контроллер WB:
- **Не используй upstream-инструкции вслепую.** `curl | sh` — это generic Tailscale install, но в WB он ставится из штатного репозитория (`apt install tailscale`).
- **Проверяй wiki.wirenboard.com** — там описана WB-специфичная процедура (свои репозитории, модули ядра, конфиги).
- **Особенности WB:** после установки может потребоваться `reboot` (подгрузка модулей ядра), настройка NAT через nftables (не iptables), специфичные пути к конфигам.
- Если вики-страница есть — используй её как источник, не upstream-документацию.

## Аудит жалоб из Google Sheets против YT

Когда пользователь просит разобрать книгу жалоб (Google Sheets) и проверить, какие записи уже есть в YouTrack:

1. **Чтение**: `gsheet_get(range='Жалобы!A12:K1400')` — все строки с данными.
2. **Фильтр**: только строки с проектом `интеграции` в колонке F (индекс 5).
3. **Отсев обработанных**: отфильтровать только те, где I (меры, индекс 8) пусто И K (решена, индекс 10) пусто. Это «необработанные».
4. **Поиск в YT**: для каждой необработанной записи — `yt_search(project='INT', text='ключевые_слова')`.
5. **Категоризация результата:**
   — **Уже есть** — тикет существует подходящий (даже если завершён)
   — **Пересекается частично** — есть смежный тикет, но запись уточняет/дополняет
   — **Чистый новичок** — ничего не нашлось
6. **Итог:** краткие списки по каждой категории с номерами строк из Sheets. Ссылки на YT — группировать в конце, не инлайнить.

- **Не все записи с F="интеграции" — про интеграции как продукт.** Некоторые — про UI (генератор шаблонов), софт, железо. Пользователь решает включать или нет.

### Обратный ход: из YouTrack в Telegram

Если search по TG не нашёл дискуссию пользователя — проверь YT-тикет на эту тему. В описании тикетов часто есть секция «Ссылка на обсуждение» с прямыми ссылками на Telegram-сообщения вида `t.me/c/<chat_id>/<msg_id>`.

**Паттерн (23.06.2026):**
1. Пользователь: «посмотри в чате что Y писал про Z»
2. `telegram_search(query=Z)` — пусто
3. `yt_search(text=Z)` — находишь тикет
4. `yt_get_issue` → проверяешь description на Telegram-ссылки
5. Из ссылок берёшь `chat_id` и `msg_id` → `telegram_thread(chat_id=..., message_id=<msg_id>)`

**Когда это срабатывает:** тикет в статусе Идея/Open — в нём линкуют исходное обсуждение. Особенно если автор тикета аккуратен.

## Формулировка тикета (для ручного ввода / черновика)

Когда пользователь просит «сформулируй тикет по теме» — не вычитывать, а дать краткую заготовку. пользователь сам решает что и как вбивать.

**Два режима в зависимости от запроса:**
1. **«Сформулируй тикет»** — краткий блок (Summary + Description, 5–10 строк), без полей-заголовков. Формат ниже.
2. **«Распиши идею / полный текст»** — структурированный формат с секциями (Суть идеи, Проблема, Охват...).

**Принцип: тикет о фиче = добавить, не «оценить».**
Если пользователь просит тикет на новую функциональность (которой нет), пиши как задачу на **добавление**, а не на исследование/оценку. Не «Оценить, поддерживает ли навык TTS» — а «Добавить TTS-ответ». Знаний о том, поддерживает или нет — у тебя нет. Если не знаешь — не пиши «оценить», напиши «добавить» — это контекст задачи, которую проектируют, а не оценивают.

**Формат:**
- Type / State / Summary / Description — всё в одном небольшом блоке
- **Summary:** одна строка, суть — устройство + проблема
- **Description:** 3-5 строк без отбивок и лишней воды. Только:
  — кто не работает / что сломалось
  — пруф (ссылка на support thread)
  — что нужно сделать (1-2 пункта)
- **Никаких:** полей с заголовками «Assignee:», «Tags:», «Контекст:», «Что нужно сделать:». пользователь сам заполнит поля в YT интерфейсе.
- **Без эмодзи**, без многословия, без ремарк-инструкций типа «(на усмотрение)»
- Максимум — 10 строк для всего куска. Если уложился в 5-7 — отлично.

**Пример правильного:**
```
INT | Bug | Open
Summary: Vakio OpenAir скрипт в wb-community — trailing commas, несовместим с Duktape/ES5

Скрипт VakioOpenair.js из wb-community не работает — wb-rules использует Duktape (ES5).
Trailing commas в массивах/объектах. После ручного удаления — заработал.
Пруф: [ссылка]
Нужно: привести к ES5 и обновить в репозитории.
```

**Пример НЕправильного** (пользователь сказал «ой как длинно короче»):
```
Проект: INT
Type: Task
State: Open
Assignee: (на усмотрение)
Tags: интеграции, pulsar, protocol, rs-485, electricity-meter

Summary: Интеграция счётчика Пульсар 1Ф4Т (протокол V1) — не опрашивается с WB 8

Description:
Пользователь не может снять данные с электросчётчика Пульсар 1Ф4Т...
[развёрнутая простыня с разделами, маркированными списками, тегами]

Нужно что-то поменять — summary, приоритет...
```

**Питфолл (15.06.2026): Не выдумывать цифры для раздела «Охват».**
Если пользователь просит расписать идею с охватом — не выдумывать проценты наобум («30-50% проектов», «каждый второй пользователь»). Только то, что можем обосновать:
- количество тем на support
- записи в книге жалоб
- упоминания в TG (с оговоркой о покрытии кэша)
- данные с WBCE/выставок
Без источника — писать «точных данных нет, по имеющимся обращениям — X».
