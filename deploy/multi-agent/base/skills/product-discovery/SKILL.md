# Product Discovery (боль, а не хотелка) + разбор конкурентов

Запрос пользователя — это **его догадка о решении**, не проблема. Задача — вскрыть боль и решать к ней.
Полезно инженерам ТП, интеграторам и разработчикам: правильно понятый запрос → правильный тикет/фича.

## Инструменты (terminal выключен)
- Клиентские боли в чатах — по уже проиндексированным сообщениям: `telegram_search` (найти), `telegram_recent` (лента чата), `telegram_thread` (размотать тред по message_id). Полной выгрузки истории по датам нет.
- Портал/веб/конкуренты → `web_search` → `web_extract`; скачать файл/картинку паспорта → `http_fetch`.
- Тикеты → `yt_*`; книга жалоб (Google Sheets) → read-тулы google_sheets; спорное сравнение A/B → делегируй сабагенту (delegation).
- Документ для стейкхолдеров сохраняй в рабочий `.md`-файл (file-тулы) и приложи как файл.

## Рефрейм: три слоя
- **Хотелка** — предложенное пользователем решение («добавьте таймер»).
- **Боль** — что на самом деле болит и почему это плохо.
- **JTBD** — работа: кто, в какой ситуации, что пытался сделать, что заблокировало.

Хорошая фича решает боль/JTBD; буквальная хотелка часто закрывает её лишь частично.

## Реконструкция боли без интервью
1. **Дойди до первичного артефакта, не до перефразировки.** Сырая жалоба клиента ближе к боли, чем триаж-тикет. «У меня X не работает» ценнее, чем «добавьте Y».
2. **JTBD-скелет:** кто (роль) · ситуация · цель · что заблокировало · **текущий костыль** · последствие. Костыль выдаёт боль — пользователь сам придумал обход.
3. **5 «почему»** по артефакту — отделяй симптом от причины.
4. **Gap-анализ: реконструкция vs оригинал.** Появился оригинальный текст клиента после пересказа — сравни явно, выпиши что реконструкция поймала, а что пропустила. Разрыв = слепые зоны, закрой их в выводе.
5. **Severity × Reach — цифрами, не на глаз.** Один клиент — ещё не паттерн. Сколько источников проверил, сколько нашлось — числом: «1 запись в книге жалоб; 3 чата (1100 сообщений/30 дней) — 0 упоминаний; у конкурентов фичи нет».
6. **Проверь существующие решения** до того, как строить новое: готовые устройства конкурентов, компоненты на DIN, альтернативы. Открой паспорта/инструкции, сравни архитектуру, тайминги, напряжения, цены.

## Classify before you validate (обязательно)
Перед тем как включить сообщение из чата в список гипотез — прочитай **полный текст** (`telegram_thread`/полное сообщение, не сниппет `telegram_search`) и классифицируй:

- Продуктовая гипотеза / идея нового продукта («нужен компактный MSW в подрозетник») → ✅ в гипотезы.
- Анализ конкурента («у AirGradient есть open-source датчик») → ❌ не гипотеза.
- Вендорское уведомление («iRidi снимает KNX-интерфейс») → ❌.
- Баг существующего продукта («MSW v.4 не виден в sprut hub») → ❌ (это баг).
- Вопрос «как сделать» → ❌.
- Feature request на существующий продукт → ⚠️ зависит от контекста.

Ошибка возникает от чтения сниппета вместо полного текста — всегда открывай полное сообщение.

## Проверка technical claims (перед техническими рекомендациями)
Утверждение о физике устройства / поведении прошивки / требовании к железу — не факт, пока не подтверждено профильным инженером (схемотехник / firmware / hardware). Нет подтверждения — удали или пометь `🤔 требует проверки у <роль>`. См. `[[research]]` (тот же протокол; урок FW-1319).

## Контракт вывода (структура, не монолог)
- **Боль** — что болит и почему плохо (со ссылками).
- **Кто / контекст** — роль, ситуация, окружение.
- **JTBD** — работа в одну строку.
- **Ограничения** — из тех-обсуждения, со ссылками.
- **Кандидаты-решения** — 2–3 варианта с трейдоффами, не одно «правильное».
- **SEV × REACH** — цифры: сколько источников проверено, сколько нашлось.
- **Рекомендация** — что делать, почему, трудоёмкость, риск.
- **Открытые вопросы / что валидировать.**
- **Уверенность** — отдельно по боли и по деталям.

Для стейкхолдеров/презентации — отдельным `.md`-файлом: TL;DR (делать/нет, в каком виде, за сколько) · Проблема (verbatim-цитаты клиента) · Конкуренты (сравнение) · Варианты с трейдоффами · SEV×REACH · Ответ скептику · Рекомендация · Риски.

### User story (для быстрой коммуникации)
`Я, как <роль>, хочу <действие>, чтобы <ценность>` (+ костыль, что делает сейчас). Пример: «Я, как интегратор, хочу заходить в веб-интерфейс на контроллере из любого браузера, чтобы настраивать оборудование удалённо, без визита и без VPN. Костыль: прошу клиента включить VPN или еду сам.» Только для пользовательских сценариев; не мешать с pain analysis.

## Разбор конкурентов
Один блок на конкурента, факты сразу (без вступлений):
1. **Что сделали** — архитектура одной фразой (mesh vs hub-and-spoke, managed vs self-hosted).
2. **Как настраивается** — реальные шаги/команды из официальной документации или вики WB (не `curl | sh` из памяти).
3. **Ограничения** — что не работает, скрытые требования, зависимости.

Порядок источников: официальная документация продукта → вики WB → форумы/репо. Не нашёл факта — «не проверял, предполагаю», не утверждай без источника.

**Питфоллы сравнения:**
- **Не смешивай подписку и API pay-as-you-go** по цене за токен — apples-to-oranges (подписка даёт кратный leverage за фикс. цену). Сравнивай подписка↔подписка или API↔API.
- **Гео-блокировки** — не утверждай «заблокировано в РФ» без свежей проверки (не старше ~3 мес.). Различай: DPI-блок протокола, блок координационных серверов, блок сайта/загрузки.
- **Провайдер скрывает абсолютные лимиты** — скажи прямо «множители без базиса ни о чём не говорят»; зафиксируй структуру планов и ecosystem-фичи вместо голых множителей.
- **VPN/mesh** — уточняй: доступ к самому устройству или ко всей локальной сети; проверь конфликт подсетей при совпадении локальных диапазонов.

## Формат ссылок (всегда)
`[ключ](url)`. Префиксы: `tg` — Telegram, `sp` — support.wirenboard.com, `ddg` — поиск, `doc` — Google Docs, иначе транслит заголовка. Каждый `[ключ]` обязан иметь `(url)` — голых ключей без URL не оставляй. Без моноширинного, без `[1]`/`[2]`.

## Анти-паттерны
- реализовать хотелку буквально, не вскрыв боль;
- ролеплей за пользователя с придуманными деталями как фактами;
- одно «правильное» решение без альтернатив;
- «severity» по громкости одного запроса, без проверки частоты;
- читать сниппет/пересказ идеи вместо оригинала;
- непроверенная физика · категоричные утверждения без документации · гипотезы как факты;
- убрал ссылку/аргумент — перепроверь весь вывод: не держался ли он на удалённом утверждении.
