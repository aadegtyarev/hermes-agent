# Research (ресёрч и реконструкция контекста)

## Когда использовать
- «Раскопай», «проведи ресёрч», «разберись», «копни глубже», «найди инфу».
- Проверить гипотезу, собрать аргументы за/против, сравнить технологии/решения.
- Восстановить историю: откуда пошла идея/решение/фича, кто её просил и зачем.
- Событие с противоречивыми нарративами.

**Цель — не «что-то нашёл», а связная нить с цитатами: исток → кому нужно → суть → решение.**

## Инструменты (в этом деплое terminal выключен)
- Поиск в вебе → `web_search` (DuckDuckGo backend), затем `web_extract` на конкретные URL. Не вызывай `web_extract` на URL «из памяти» — сперва найди актуальный адрес.
- Скачать файл/картинку, дёрнуть REST/raw-URL → `http_fetch`. GitHub (репо, дерево, файлы, issues, PR) → тулы `github_*` (или `http_fetch` на `api.github.com` / `raw.githubusercontent.com`).
- Сравнить версии/файлы (аналог `diff`), распарсить, посчитать → `code_execution` (Python: `difflib`, `json`, `re`). Локального `diff`/`curl` в шелле нет.
- История чатов — **только уже проиндексированные сообщения**: `telegram_recent`, `telegram_search`, `telegram_thread`. Полную выгрузку истории по датам бот не умеет — работай с тем, что есть, и отмечай охват.
- Тикеты → `yt_*` (YouTrack). Google Docs/Sheets → read-тулы. Прошлые сессии → `session_search`. Долгая память → memory-тулы.
- Большой объём данных в одной сессии → логируй в рабочий `.md`-файл (file-тулы), чтобы не потерять при компактификации контекста.

## Принципы (обязательны)
1. **Факт — с источником.** Каждое неочевидное утверждение сопровождается ссылкой. Без источника — помечай `🤔` (догадка).
2. **Не додумывай.** Нет данных — пиши «не найдено», не экстраполируй. **Отсутствие в поиске ≠ отсутствие в реальности:** формулируй «не найдено публичных данных», не «этого нет».
3. **Сначала широко, потом глубоко.** 3–5 параллельных запросов с разных углов, затем чтение топ-источников.
4. **Контр-аргументы обязательны.** После сбора подтверждений — отдельный раунд поиска сильнейших опровержений. Назови самое сильное. Без этого вывод skewed (confirmation bias).
5. **Единая метрика.** Сравнивая — приводи всё к одной единице ($/M токенов с $/M токенов, не токены с подпиской). Subscription ≠ API pay-as-you-go.
6. **Первоисточник — базовая линия.** Для событий: сначала официальное заявление, потом медиа, потом комментарии.

## Метод
1. **Уточни вопрос и критерий «достаточно».** Что нужно: факт / хронология / решение+обоснование / «кто и почему»? От этого — глубина. Зафиксируй, что считается полным ответом, иначе остановишься рано или будешь копать бесконечно.
2. **Источники и охват — ДО поиска.** Перечисли, где может лежать ответ, и для каждого пойми, что реально доступно (период, полнота). Часть → сперва расширь охват. Требует аутентификации (YouTrack, закрытый сайт, Google Doc) — отметь `⚠️ источник не открыт`.
3. **Точки входа — широко, по синонимам.** Ищи семантическое поле: синонимы, смежные термины, коды/номера. Поиск по индексу/серверу даёт **ложные отрицания** — если должно быть, а пусто, перепроверь иначе (по дате/автору/прямым доступом). Собери якоря: идентификатор + время + автор.
4. **Размотай цепочку от каждого якоря.** Читай не только якорь, но и окружение; иди по связям (реплаи, треды, «см. …») **вверх** — к истоку, **вниз** — к развязке.
5. **Исток и развязка.** Исток — самое раннее звено (часто отсылка вовне). Развязка — где приняли решение / закрыли / оборвалось. Между ними — кто, что, почему.
6. **Кросс-источник.** Нашёл только отсылки («см. тикет», «писали в поддержку») → иди в первоисточник и свяжи: заказчик ↔ заявка/тикет ↔ обсуждение ↔ решение. Первоисточник недоступен → триангулируй из вторичных (скриншоты, пересказы), уверенность `⚠️`, укажи барьер.
7. **Синтез с цитатами.** Ответь ровно на исходный вопрос. Каждое утверждение — со ссылкой `[ключ](url)`. Явно раздели: ✅ подтверждено · ⚠️ вне охвата / не дочитано · 🤔 допущение.
8. **Критика полноты (петля).** Какой источник не открыл? какой синоним не искал? какая ветка без ответа? Есть пробел → вернись к шагу 3. Останавливайся по критерию из шага 1.

## Ранги источников
- **А:** официальные документы, API docs, arXiv, пресс-релизы, спецификации.
- **Б:** авторитетная журналистика с named sources.
- **В:** блоги, форумы, Hacker News, Reddit — с пометкой.
- **Не использовать:** пропагандистские источники, соцсети без верификации.

Формат ссылки в ответе: `[Заголовок](url) — фраза из источника`. Без моноширинного кода, без `[1]`/`[2]`.

## Проверка technical claims (перед финализацией)
Если в результате есть утверждение о **физике устройства** (нагрев, износ, токи, напряжение на катушке реле), **поведении прошивки** («код переиспользуется», «логика как у X») или **требование к железу** (новые компоненты, ревизия платы):
1. Выдели все такие утверждения.
2. Определи, кто должен подтвердить (схемотехник / firmware / hardware).
3. Нет подтверждения — не включай как факт: удали или пометь `🤔 требует проверки у <роль>`.

**Урок FW-1319:** «кран под напряжением — реле 24/7 греется» → на NC-контакте катушка обесточена. Вопрос схемотехнику снял бы ошибку за 2 минуты. Непроверенная физика, категоричные утверждения без документации и гипотезы-как-факты ослабляют документ сильнее, чем их отсутствие.

## Портал поддержки WB (нюанс)
При поиске на `support.wirenboard.com` отличай **запрос фичи** от **вопроса по подключению**: если клиента устроил ответ — это не запрос фичи, не учитывай в оценке спроса. Тема, добавленная инженером WB в книгу предложений, — запрос фичи.

## Анти-паттерны
- искать до проверки охвата источника;
- верить поиску-по-индексу без перепроверки другим способом;
- «нашёл точку входа» ≠ «понял тему» (без разматывания связей);
- отвечать без ссылок на конкретные звенья;
- молча проглатывать «результат обрезан» вместо сужения запроса;
- смешение единиц измерения в одном сравнении;
- перед фундаментальным рассуждением о проекте — сперва проверь project docs (grep/read), а не рассуждай с чистого листа.

Дефолтный объём вывода — **конденсат** (3–7 строк: суть, ключевой конфликт, вердикт, ссылки). Разворачивай только по просьбе «подробнее».
