# Telegram-контекст (плагин `telegram-context`)

Bot API не отдаёт историю — плагин **накапливает** входящие сообщения и даёт их читать.
Доступно только то, что бот видел с момента запуска (бэкфилла старой истории нет).

## Тулы
- `telegram_recent(chat_id?, limit)` — последние сообщения чата по времени. По
  умолчанию — самый активный чат. Отсюда бери `message_id`.
- `telegram_thread(message_id, chat_id?)` — собрать тред: цепочка reply вверх до
  корня + все ответы ниже, хронологически, кто→кому.
- `telegram_search(query, chat_id?)` — поиск подстроки по накопленному.
- `telegram_dm_allowlist()` — кто попал в DM-allowlist (собран из work-чатов).

## Как устроен гейтинг (не тул, а поведение)
- **work-чаты** (`TELEGRAM_WORK_CHATS`): бот отвечает; кто там пишет — автоматически
  получает право писать боту в личку.
- **read-only чат** (`TELEGRAM_READONLY_CHATS`): бот читает для контекста, но **не
  отвечает**; его участники DM-доступ НЕ получают.
- **личка**: бот отвечает только тем, кто в allowlist (участник work-чата,
  проверяется по членству, + `TELEGRAM_DM_EXTRA_USERS`).
- Добавили бота в чужой чат — он его **игнорирует**.

## Паттерн для разбора
1. Пришёл вопрос/жалоба в чате → `telegram_thread(message_id)` собрать весь тред,
   понять кто что сказал и хронологию.
2. Нужен более широкий контекст → `telegram_recent` / `telegram_search`.
3. Дальше — обычный разбор (ssh на контроллер, поиск по коду/форуму и т.д.).
