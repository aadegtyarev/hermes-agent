# Diagnostic Cross-Weaving

## Когда использовать

Любая проблема, где есть публичные источники:
- Ошибка в чате / сообщении пользователя
- Жалоба на сервис в группе/канале
- Инцидент в мониторинге
- Тикет в поддержку
- **Жалоба/баг, совпадающая по времени с релизом или деплоем** — в первую очередь смотри релизы и release notes, а не код

**Не применимо:**
- Проблемы с доступом к серверу, где нужны логи
- Внутренние баги с кодом (там `systematic-debugging`)

## Процесс

### Фаза 0: Загрузить скиллы (обязательно, не пропускать)

**Перед первым tool call** — остановись и просканируй список доступных скиллов в system prompt. Если тема задачи хоть частично совпадает с описанием скилла — загрузи его.

Это **не** рекомендация, а механическое правило, которое нарушается систематически. Причина сбоя: задача выглядит знакомой → хочется сразу делать → пропускается шаг сканирования.

Сопоставление по типу задачи:
- Диагностика проблемы → `diagnostic-cross-weaving`, `systematic-debugging`
- Задача с кодом → `codebase-inspection`
- Работа с репозиторием / баг-трекером → `github-issues`
- Анализ чатов/данных → загрузить соответствующий skill **до** начала сбора данных

### Фаза 0.5: Сформулировать проблему (уже со скиллами в контексте)

Прежде чем что-то искать — запиши одной фразой: **что именно сломалось, для кого, когда**.

**Плохо:** «что-то с сертификатами»
**Хорошо:** «nginx выдаёт 502 при проксировании на upstream:8042, в логах certificate chain too long, проблема началась 18 июня после перевыпуска сертификата»

Если формулировка размытая — уточни, пока не станет конкретно.

### Фаза 1: Собрать крупицы — три линии

#### Линия A: Код / конфигурация

Где живёт логика проблемного узла:
- Репозиторий пакета
- Скрипты генерации конфигов
- Шаблоны nginx/apache/whatever
- Поиск по ключевым словам: `proxy_ssl_verify`, `proxy_pass`, `verify_depth`

**Читай буквально, не экстраполируй.** Если в скрипте нет `verify_depth` — это факт. Не додумывай «наверное там дефолт такой-то».

#### Линия B: Текущее состояние endpoint

Проверь, как система работает прямо сейчас:
- Для HTTPS: `curl -v https://host:port/ 2>&1 | grep -i "certificate level\\|issuer\\|subject"`
- Для API: какой HTTP-статус, что в теле
- Для DNS: резолвится ли, какие IP
- Для TCP: открыт ли порт
- **Для UI-логов:** если пришёл скриншот дашборда/лог-вьювера — прочитай весь скриншот внимательно, затем проверь соответствующую линию (agent.log, gateway.log) напрямую. Скриншот — это **подсказка куда смотреть**, а не отмена проверки в сырых логах. Таймзона в UI может отличаться от таймзоны логов — учитывай сдвиг.

**Не предполагай — проверь.** Если нет openssl, curl всё покажет.

#### Линия C: Интернет / чаты / GitHub / доки

Ищи в порядке вероятности:
1. **Ошибку дословно** — 'certificate chain too long' nginx proxy
2. **GitHub Issues** пакета — есть ли свежие баги
3. **GitHub Issues смежного проекта** — если проблема системная
4. **GitHub Releases** — если проблема совпала по времени с релизом, это первая гипотеза
5. **Официальные доки** — nginx.org, letsencrypt.org/certificates/
6. **Блоги/анонсы** — release notes, GitHub changelog
7. **Чатики / форумы** — другие пользователи с той же ошибкой

**Не ленись читать.** Если ссылка ведёт на первоисточник — прочитай целиком. Там может быть точная формулировка, объясняющая всё.

### Правило: первоисточники, не пересказы

В Линии C — если нашёл объяснение проблемы на вторичном источнике (агрегатор новостей, блог-пересказ):

1. **Найди оригинальный документ**, на который ссылается статья — и прочитай его.
2. Ссылайся в отчёте на **первоисточник**, а не на статью.
3. Если статья говорит «правила CAB Forum изменились» — иди на cabforum.org/ballots/, найди конкретный баллот/секцию, вытащи цитату.
4. Если говорит «CA обновил соглашение» — иди в его repository, найди документ.

Исключение: статья содержит **прямую цитату из письма/документа**, которого нет в открытом доступе. Тогда цитируй статью как носитель цитаты, но добавь: «по данным [издание], письмо [должность] от [дата]».

Пример правильной цепочки:
```
Плохо: «Пишут, что CA/B Forum ввёл санкционные проверки»
Хорошо: «§3.2.2.12.2 EV Guidelines v2.0.2 от 4 мая 2026: CAs must check OFAC SDN, BIS Denied Persons, EU equivalents при выдаче»
  + cabforum.org/working-groups/server/extended-validation/guidelines/
```

### Фаза 2: Свести в кучку

Когда по каждой линии есть факты — попробуй собрать нарратив.

**Правило трёх линий:** минимум два независимых источника, сходящихся к одной причине. Идеально — три.

Пример:
```
A: configure-nginx-proxy.sh — нет verify_depth, дефолт 1
B: curl upstream:8042 — 4 уровня (3 промежуточных CA)
C: letsencrypt.org/certificates/ — YR цепочка: EE→YR→Root YR→ISRG Root X1
  + github.com/nginx/nginx/issues/1409 — тикет о дефолте depth=1
```

Все три линии → «nginx не хватает глубины верификации для новой цепочки LE».

### Фаза 2.5 (разветвление): Расследование выкатки/релиза

Когда проблема совпала по времени с обновлением пакета или деплоем:

1. **Засеки дату первого сообщения о проблеме** — из чата, тикета, мониторинга
2. **Проверь даты релизов** — GitHub Releases, changelog
3. **Сравни версии** — release notes, diff между тегами
4. **Проверь архитектурные изменения** — не багфикс, а замена компонента (смена схемы доставки, замена коллектора)
5. **Свяжи с тикетами** — feature закрыта, но реализация ушла в другой проект; баги с похожим симптомом
6. **Найди несоответствие документации коду** — продуктовый документ описывает одно, код реализует другое

**Симптомы rollout-проблемы:**
- «данные то появляются, то пропадают» → сброс state-file / двойная запись / новая таблица
- «раньше работало, после обновления сломалось» → изменился формат или протокол
- «на старых устройствах ок, на новых нет» → агент обновлён не везде

### Фаза 3: Проверить гипотезу

Если гипотеза даёт предсказание — проверь:
- «Если добавить verify_depth 5 — заработает» → пользователь подтвердил
- «Если откатить агент до предыдущей версии — данные стабильны» → подтверждает rollout-гипотезу

Не выдавай гипотезу за факт, пока не проверил.

### Фаза 4: Доложить

Формат:
1. **Формулировка проблемы** (одна строка)
2. **Что нашли** (по линиям, со ссылками)
3. **Корень** (одна фраза «потому что…»)
4. **Фикс** (что делать, желательно с командой)
5. **Системное решение** (что надо поменять в коде/репозитории, чтобы не повторилось)

#### Формат для каскадных багов (multi-component)

Когда проблема затрагивает больше одного пакета/компонента — подавай как **цепь**:

```
[Компонент A: причина] → [Б: ускоритель] → [В: вторичный эффект] → [Г: наблюдаемый симптом]
```

Для каждого звена укажи:
- **Где** — пакет/репозиторий/компонент (со ссылкой на код: строка)
- **Что конкретно** — цитата из кода или тикета
- **Тип бага** — primary cause / accelerator / secondary / symptom
- **Статус фикса** — есть ли тикет, PR, workaround

Блок «Расхождения с данным диагнозом» (таблица): если диагноз от первого респондента не совпадает с фактической причиной, покажи это явно — не в тексте, а таблицей.

#### Доставка артефактов

**Принцип:** результаты длительного анализа (досье, отчёты, скрипты, логи) — файлами/архивами, а не сырым текстом в чат.

**Когда упаковывать в файл:**
- Анализ больше 50 строк — пиши в файл, отправляй ссылкой на файл
- Есть код/скрипты/логи — архив (zip/tar.gz) с подпапками
- Много связанных файлов — раздели на логические архивы

**Структура багрепорта:**
```
bugreport/
├── README.md    — TL;DR, команды воспроизведения
├── patches/     — файлы патчей (*.patch)
├── scripts/     — скрипты воспроизведения
├── logs/        — сырые логи экспериментов
├── diagrams/    — схемы
└── deb/         — собранные .deb пакеты (если есть)
```

**Защита от context compaction:** промежуточные результаты длительного (>5 tool calls) анализа — пиши в файл немедленно. Не храни всё в контексте: как только получил значимый факт/метрику/вывод — сохраняй в лог-файл сессии.

**Распределение файлов между агентами:** если нужно поделиться скиллом/методикой с другим агентом — сохрани файл в общую директорию или отправь пользователю, который перешлёт. Не предполагай, что у другого агента есть доступ к твоей файловой системе.

### Правило разделения фактов и гипотез

Фразы вроде «с наибольшей вероятностью» или «классические последствия» — это fabrication, если ты не проверил конкретный механизм.

**Формат доклада с неопределённостью:**
1. **Точно знаю** — факты из кода, логов, релизов, тикетов. Со ссылками.
2. **Предполагаю, не проверял** — гипотезы без подтверждения. Каждую с пометкой 🤔.
3. **Опровергнуто** — что проверил и не подтвердилось.

### Формат заземлённого доклада (уровни уверенности)

Используй 🟢/🟡/🔴 уровни уверенности в сводных таблицах — этот формат читается заметно лучше текстовых описаний:

| Метка | Значение | Когда ставить |
|---|---|---|
| 🟢 | Доказано экспериментом или первичным источником | Ты сам воспроизвёл, прочитал код, замерил, проверил команду |
| 🟡 | Из первичных данных клиента/стороннего источника | Ты видел данные, но не можешь воспроизвести без его среды |
| 🔴 | Не установлено / нужны дополнительные данные | Гипотеза, не проверенная ни одним методом |

**Правила:**
- Маркируй каждый значимый вердикт в итоговой таблице. Не ставь 🟢 «на глаз» — если не проверял лично, максимум 🟡
- Если потом подтвердил — подними уровень (🔴→🟡→🟢). Если опроверг — 🔴 с явной пометкой «опровергнуто»
- Используй именно в сводной таблице доказательств, где каждая строка — одно утверждение и его статус. В нарративе достаточно внутритекстовых пометок вроде «(🟢 доказано)»

Пример сводной таблицы:

| Утверждение | Статус |
|---|---|
| Аварийный уход брокера → зависимый сервис мёртв до reboot (BindsTo) | 🟢 доказано (kill -9, воспроизводимо) |
| memory_limit роняет брокер по OOM | 🟢 опровергнуто (предохранитель, брокер жив) |
| 30limits.conf — причина регрессии | 🟢 отклонено (не менялся с 2024) |
| Что уводит брокер в inactive у клиента | 🔴 не установлено (нужны journal/dmesg) |

Пример разделения факт/гипотеза:
```
Факты:
- v1.8.1 от 15 июня — смена схемы доставки скрипта (шаблон локально, переменные с облака)

Предположения (не проверено):
- 🤔 State-file мог сброситься при смене формата доставки → проверил: нет, путь задаётся агентом, не менялся
- 🤔 Пустые переменные на старте → проверил: читается сохранённый конфиг, пустого скрипта не будет

Опровергнуто:
- State-file не сбрасывается — агент сам задаёт путь
- Пустых переменных не бывает — конфиг сохраняется и перечитывается
```

Не выдавай гипотезу за факт, пока не проверил и не подтвердил проверкой.

## Инструменты

| Что нужно | Что использовать |
|-----------|-----------------|
| Смотреть исходное сообщение в чате | контекст переписки вокруг сообщения |
| Искать в интернете | web search + extract |
| Читать файлы на GitHub | extract по URL (raw или GitHub web) |
| Проверить endpoint | `curl -v host:port` в терминале |
| Искать в коде | поиск по файлам / GitHub поиск |
| Официальные доки | extract по URL из поиска |

## Pitfalls

- **Не подставляй URL из памяти.** Ссылки протухают. Каждый раз сначала поиск, потом extract.
- **Не додумывай детали.** Если в скрипте нет директивы — это факт «нет», не «наверное дефолт 1».
- **Одна линия ≠ корень.** Ошибка в чате + похожий GitHub issue может быть совпадением. Нужна вторая/третья линия.
- **Не выдавай fabrication.** Если не можешь проверить — скажи «не проверял, предполагаю».
- **Проверяй премиссу гипотетического вопроса.** На «а что если поменять модель компоненту X?» не строй нарратив, пока не убедился, на какой архитектуре компонент живёт. Код склонирован — grep за 10 секунд покажет. «А что если у нас в проде…» — сначала проверь, что в проде, а не по памяти.
- **Не тащи stale context между сессиями.** Если помнишь имя компонента/плагина из прошлой сессии — проверь, что он всё ещё существует, прежде чем на него ссылаться. Первую минуту трать на «что сейчас в системе», а не «что я помню из прошлого раза».
- **Не утверждай отсутствия без проверки.** «У X нет Y» — если код/репозиторий доступен, grep за 10 секунд. Не склонирован — спроси, не додумывай. Уверенный «у X нет Y» при доступном репозитории — fabrication.
- **Если тебя поправили — не защищайся, проверь сразу.** Первая реакция на коррекцию — верификация («гляну код»), а не контраргумент.
- **Когда пользователь указал путь — execute, не re-research.** Если он сказал «probe X сейчас» — сделай probe X или скажи почему нельзя (нет доступа, поле отсутствует, система не запущена). Не уходи в исследование того же вопроса с другого угла — это выглядит как зацикливание. Если после 2 поисков по теме не нашлось нового — остановись и докладывай.
- **Не экстраполируй ограничение с одной инстанции на другую.** Одна и та же логика на разных развёртываниях (разные модели, разные хранилища) может вести себя по-разному. Проверяй конкретную инстанцию.
- **Сперва — что есть, потом — что значит.** Сначала факты, потом интерпретация.
- **Не экономь на чтении.** «Пробежался глазами» → пропустил ключевую деталь. Читай целиком.
- **Не доверяй первому диагнозу.** Инженер/саппорт может дать верный workaround, но неверную причину. Если сказано «X — причина», проверь: открой код X, прочитай исходник. Workaround может лечить симптом, а корень — в соседнем пакете.
- **Если сомневаешься в выводах — привлеки сабагента с более мощной моделью.** Передай ему все собранные факты и попроси проверить логику или найти слабые места в гипотезе. Не используй для сбора фактов (он без памяти сессии), только для анализа/верификации.
- **Скиллы надо загружать ДО первого tool call, а не после.** Если понял, что работаешь без скилла — остановись, загрузи, перепроверь план.
- **Промежуточные результаты — пиши в файл сразу, не держи в контексте.** При длительном анализе (5+ tool calls) context compaction может стереть промежуточные выводы. Как только получил значимый факт/метрику/вывод — пиши в лог-файл сессии.
- **После завершения задачи — предложи сохранить методику как скилл.** Если работа закрыла новый повторяющийся паттерн — предложи создать/обновить skill.

## Техника форсированного доказательства (proactive probing)

Когда проблема не воспроизводится естественно (медленный рост, редкие условия, длинный цикл):

### Алгоритм

1. **Понять механизм из changelog/upstream fix.** Найди точное описание бага — не «утечка памяти», а «mismatched alloc/free в properties». Определи, какой именно код/путь был исправлен.
2. **Спроецировать на целевую систему.** Есть ли на целевом устройстве та же версия с багом? Какой код/путь триггерит баг?
3. **Сконструировать probe — минимальный ввод, который гарантированно проходит через faulty code path.** Не имитируй полную нагрузку — найди одиночное действие, которое зажигает баг. Пример: не 3 дня реальной работы, а 1 MQTTv5 PUBLISH с properties.
4. **Измерить артефакт, а не симптом.** Симптом = OOM (через 3 недели). Артефакт = расхождение между счётчиком и реальностью (через 1 минуту). Найди метрику, которая показывает баг напрямую.
5. **Сравнить с эталоном.** Если артефакт ведёт себя ровно так, как описано в баг-репорте — механизм доказан, даже если полный сценарий ещё не развернулся.

### Пример: mosquitto heap accounting bug (#3192)

- **Проблема:** пользователь жалуется на OOM mosquitto через 3+ дня работы
- **Changelog:** 2.0.21 чинит «mismatched wrapped/unwrapped memory alloc/free in properties»
- **Probe:** 1 MQTTv5 PUBLISH с properties — должно увеличить `$SYS/broker/heap/current`, но не затронуть реальную RSS
- **Результат:** после 10K PUBLISH: HEAP +253K, RSS 0 — баг подтверждён за 2 минуты вместо 3 недель
- **Ключ:** probe не ждёт естественного течения — он доказывает механизм, зная куда смотреть

### Когда применять

- Баг описан в changelog upstream, но не портирован в текущую версию
- Воспроизведение в проде занимает часы/дни
- Есть контролируемая среда (тестовый стенд)
- Известен механизм бага (какой код/паттерн его триггерит)

### Шаг 5 (опционально): A/B тест — доказательство фикса

После того как probe подтвердил баг — **запусти тот же probe на версии с фиксом** и покажи, что симптом исчезает. A/B тест — самое убедительное доказательство:

|  | Без фикса | С фиксом |
|---|---|---|
| **Probe** | 10K MQTTv5 PUBLISH c properties | Тот же probe |
| **Симптом** | HEAP +253K, RSS 0 — счётчик врёт | HEAP +2K (разовая инициализация), RSS 0 — норма |
| **Вывод** | Баг воспроизведён | Баг исправлен |

Алгоритм:
1. Найти/собрать/запустить версию с фиксом (следующий upstream релиз, бэкпорт коммита, testing-репозиторий)
2. Убедиться, что probe задевает тот же code path (тот же протокол, те же параметры)
3. Прогнать probe, замерить те же метрики
4. Сравнить результат с «до фикса» в таблице

### Питфоллы

- **Не путай probe с эмуляцией полной нагрузки.** Эмуляция пытается повторить реальный сценарий. Probe проверяет конкретный механизм. Если probe работает — механизм доказан, эмуляция не нужна.
- **Проверь, что probe задевает именно faulty code path.** MQTTv3 publish не заденет баг #3192 — нужен именно v5 с properties.
- **На A/B тесте probe должен быть идентичным.** Разные параметры = несравнимые результаты.

## Метод: Верификация чужого анализа (dossier cross-check)

Когда тебе передают чей-то готовый анализ/досье/отчёт и просят сверить с твоими выводами или с реальностью.

### Зачем

Даже качественный отчёт может упустить ключевую гипотезу — не потому что автор плох, а потому что он проверял то, что пришло в голову, а не то, что оказалось правдой. Твоя задача — найти то, что автор **не** проверял.

### Процесс

**1. Прочитай отчёт целиком.** Составь таблицу всех утверждений автора:

| Утверждение | Статус автора | Твой статус |
|---|---|---|
| ... | 🟢/🟡/🔴 | ? |

**2. Для каждого утверждения — проверь независимо.** Не принимай на веру даже 🟢-марку автора:
- Если «из эксперимента» — повтори мысленно: «какие ещё причины могли дать тот же результат?»
- Если «из кода» — открой ту же строку и прочитай сам
- Если «из данных клиента» — можешь ли получить те же данные из другого источника?

**3. Найди гипотезы, которые автор НЕ проверял.** Для каждой гипотезы спроси: «А есть ли альтернатива, которую автор не рассматривал?»

**Классический приём:** возьми **каждый фактор**, который автор упоминает как стабильный/неизменный, и спроси — «а когда этот фактор начал действовать?». Пример:
- Автор: «30limits.conf не менялся с 2024 → значит не причина регрессии»
- Ты: «Файл не менялся. А include_dir, который его активирует, когда добавили?»
- Ответ: скрипт начал добавлять include_dir в мае 2025 — истинная причина регрессии

**4. Для каждой «слепой зоны» — заземли:**
- Что именно не проверено (цитата из отчёта)
- Почему это могло быть причиной (механизм)
- Каким экспериментом/командой можно проверить
- Результат проверки (если выполнил)

**5. Итоговая таблица расхождений:**

| Утверждение отчёта | Статус автора | Твой вердикт | Основание |
|---|---|---|---|
| «memory_limit роняет брокер» | 🟢 | 🔴 опровергнуто | Эксперимент: kill -9, брокер жив, отключает только клиента |
| «30limits.conf не менялся → не причина» | 🟢 | 🟡 верно, но неполно | Файл не менялся, но include_dir добавили (скрипт, май 2025) |
| «Причина регрессии не установлена» | 🔴 | 🟢 найдена | commit — скрипт начал добавлять include_dir в mosquitto.conf |

### Питфоллы

- **Не соглашайся с автором «по умолчанию».** Автор мог не проверить то, что кажется очевидным. Проверяй каждое утверждение.
- **Одна «слепая зона» ≠ ошибка автора.** Автор сделал добросовестный анализ, но не додумался до конкретной гипотезы. Твоя задача — не осудить, а дополнить.
- **Начинай с того, что автор считает стабильным/неизменным.** Именно там чаще всего лежит истинная причина, которую «все знают» и никто не проверяет.
- **Проверяй не только что изменилось, но и что начало применяться.** Файл конфига мог не меняться годами, но механизм его подключения (include_dir, override, скрипт инициализации) появился заново. Разница между «файл существует» и «файл читается» — ключевая слепая зона.
- **Ищи многослойные причины.** Проблема может быть: баг в mosquitto (#3192) → но проявляется только под memory_limit → который активируется скриптом → который входит в пакет конфигов → который обновился вместе с прошивкой. Каждый слой — отдельная гипотеза.
- **Regression dimension mismatch — тестируй правильную ось.** Автор проверил «изменился ли код?» (нет) и «изменилось ли поведение?» (нет) → заключил «причина не заземлена». Но реальное изменение было в **экосистемном контексте**: та же версия шла с одним дистрибутивом (memory_limit=0), а новая — с другим (memory_limit=100M). Код не изменился, но окружение изменилось. Признак mismatch: автор проверил A и B, обе неверны, но не рассматривал третий фактор.

## Принцип: системное, не поведенческое

Если обнаруживаешь повторяющуюся ошибку в своём поведении — ищи **механическое принуждение**, а не «буду стараться». Обещание «делать паузу» не работает, если ты забываешь её делать.

В Hermes есть `pre_tool_call` hook — регистрируется в плагине (`~/.hermes/plugins/<name>/`), запускается перед каждым tool call, может **заблокировать** вызов с сообщением. Пример: блокировать `ssh_run`/`web_search`/`read_file`, пока в текущем сеансе не был загружен ни один скилл.

Это предпочтительный подход к самокоррекции: не записывать в память «буду делать X», а ставить блокировку, которая не даст сделать не-X. Детали — `references/hermes-pre-tool-call-hook.md`.

## Расширение: верификация архитектуры через code-probe (сверка дизайна с кодом)

Когда задача — **проверить, соответствует ли код архитектурным документам** (не найти баг, а верифицировать, что заявленное поведение реализовано, и обнаружить расхождения):

### Когда применять

- Пользователь говорит «сделай probe / запусти recall / проверь X прямо сейчас» — execute или чётко скажи, почему нельзя, и предложи альтернативу. Не уходи в re-research того же вопроса.
- Документ говорит «реализовано X» — нужно подтвердить file:line
- Документ описывает «дефект #N» — проверить, не починен ли уже в коде
- Подозреваешь, что код живёт не по spec'е (делает больше/меньше/иначе)
- Нужно найти gap между документированным поведением и фактическим

### Процесс

**1. Выбери единицу трассировки** — один data item, который проходит через систему. Что именно берётся на вход и где ожидается на выходе.

**2. Найди точки трансформации (lifecycle).** Пройди код от создания до финального выхода. Для каждого этапа:
- **Точка входа** — где data item рождается (тип, параметры, confidence, source_type)
- **Обработка** — кластеризация, фильтрация, quality gate, LLM call
- **Хранение** — куда записывается, какие колонки, какие значения по умолчанию
- **Recall** — как читается, какие фильтры, что возвращается

**3. В каждой точке — три вопроса:**
- Что документ обещает?
- Что код делает?
- **Расхождение?** (file:line для обеих сторон)

**4. Ищи четыре типа outcome:**

| Тип | Признак |
|-----|---------|
| **BUILT** | Code реализует то, что документ называет «design/unimplemented» — документ устарел, код впереди |
| **DESIGN (gap)** | Документ обещает, код не реализует — реальная дыра |
| **SURPRISE** | Код делает больше/иначе, чем документ описывает |
| **ERASURE** | Данные о происхождении есть в хранилище, но read-path их не читает — активное затирание на выходе |

**5. Задокументируй каждое расхождение — file:line.** Если расхождений нет — это тоже результат: «подтверждено на строке N».

Для более широкого аудита (не один data item, а вся кодовая база на классы архитектурного долга) — см. `references/architectural-debt-audit.md` (6 probe-категорий с grep-командами и критериями вердикта).

### Питфоллы

- **Не верь комментариям — читай код.** TODO может говорить «отложено», а код уже фикс.
- **Документ живёт своей жизнью.** Код может быть починен между версией документа и текущим тэгом. Всегда проверяй код.
- **Смотри default values.** `None` по умолчанию может значить «не передано → NULL в БД», а на read-path это превращается в принудительную перезапись. Default-значения — часть архитектуры.
- **Проверяй schema таблиц.** Колонка может физически отсутствовать, даже если на read-path есть попытка её прочитать.
- **Трассировка в обе стороны.** Не только «от рождения к recall», но и от recall к источнику.
- **Три точки минимум.** Одна рассинхронизация — случайность. Две — паттерн. Три — архитектурное свойство.

## Фаза 5 (опционально): Синтез общего принципа из N доменов

Когда пользователь даёт материалы из разных, несвязанных доменов (код, психология, организационный конфликт) и просит сформулировать единый принцип / контракт / фокус:

1. **Пойми каждый домен отдельно** — одна строка сути на домен
2. **Найди изоморфную структуру** — например «утверждение не несёт доказательства» во всех трёх
3. **Сформулируй принцип одной фразой** — «Earned, not stamped» / «No facade» / «Каждое утверждение несёт собственное доказательство»
4. **Покажи каждый домен как instance того же принципа** — таблицей: домен → как было (stamped) → как должно быть (earned)
5. **Закончи actionable implication** — что реально меняется в коде/поведении

**Питфоллы:**
- Не перекодируй второй домен в термины первого (человеческая психология → отвечай про человека, потом bridge к коду)
- Изоморфизм ≠ одинаковость — форма общая, content разный
- Не принуждай конвергенцию — если параллель натянута, скажи честно
- Принцип без «что делать» = пустота — всегда concrete следствие

**Сигнал смены рамки — «непонятно» от пользователя:** если после синтеза пользователь говорит «непонятно, как это относится к [домен X]» — значит ты ответил не в той рамке. На вопрос про человеческую психологию нельзя начинать с кода. Стоп, переформулируй: ответь сначала в рамке вопроса, потом bridge к коду одной фразой.

**Валидация синтеза через более мощную модель:** после формулировки общего принципа — запусти сабагента на более мощной модели для независимой проверки. Отдай ему: (1) сырые данные из каждого домена, (2) сформулированный принцип, (3) таблицу домен→stamped→earned. Попроси: «найди слабые места, перегибы, что упущено».

**Источник-картинка (screenshot статьи):** если один из доменов пришёл как изображение — сначала прочитай содержимое (vision), и только потом включай в синтез. Не пересказывай картинку по памяти.
