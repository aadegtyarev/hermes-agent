# Proactive Failure Mapping

_Техника: взять известный AI agent failure, разобрать механизм отказа, спроецировать на целевую архитектуру._

## Когда использовать

- Пользователь показывает интерес к конкретному failure case
- Пользователь спрашивает «это про нас?»
- Пользователь описывает архитектурную дыру (нет circuit breaker, нет kill switch) и хочет понять последствия
- При проектировании нового компонента — проверить на известные failure modes до того, как они случатся

## Формат разбора (4 минимальных вопроса)

Любой failure case разбирать по этим осям:

1. **Цепь ретраев: в тот же сервис или каскад?**
   — Ретраи в тот же сервис = amplification (один fail → N retries → каждый ретрай加重ляет нагрузку)
   — Ретраи через chained downstream = cascading spiral (oдин fail → retry → downstream загружен → ещё fail → ещё retry)
   — Экспоненциальный backoff не всегда спасает: он может быть на каждом уровне, но цепь не скоординирована

2. **Почему мониторинг не остановил?**
   — Пороги сработали слишком поздно (x10 от baseline → уже горело)
   — Алерт шёл человеку, не automation action
   — CloudWatch alarm → PagerDuty → on-call с ноутбуком → $30K сгорело за время реакции
   — Spending cap либо не стоял, либо стоял soft (notify, не throttle)

3. **Где должна была стоять защита и почему её не было в спецификации?**
   — Circuit breaker между сервисами (размыкание при N ошибок → dead letter queue)
   — Hard spending cap c automation action (throttle API, concurrency limit)
   — Error budget в цепи ретраев (X ошибок/мин → стоп ретраев на Y минут)
   — Почему не было: «на малом трафике не нужно», «DevOps не дали права», «бизнес-критичный — не можем остановить»

4. **Проекция на архитектуру пользователя**
   — Что из этой цепи есть у пользователя (retry, cascade, single budget line)
   — Чего нет (circuit breaker, hard cap, kill switch)
   — Сколько сгорит при таком же сценарии (токены × цена)

## Проекция — конкретные цифры

Не «у тебя тоже может случиться». Посчитай:
- Если у пользователя средний запрос N токенов, цена M/1M
- 10M invocations × N токенов = сколько сгорит с его кэш-хитом
- Сравни с его дневным бюджетом: сколько «дней жизни» сожжёт один сценарий

Пример проекции кейса $75K Lambda Spiral на агента с бюджетной LLM:
- Средний запрос ~1K токенов, ~$0.40/1M input (cached)
- 10M invocations × 1K = 10B tokens → $4,000 без кэша
- С кэш-хитом 97% → ~$120
- При бюджете ~$2/день: 60 дней сожжёт один такой сценарий

## Проекция — структурная уязвимость

Не «у тебя нет circuit breaker» — а:
- [ ] Spending cap на уровне вызова? (не $N/день, а $X/сессия)
- [ ] Retry с затуханием по времени? (2^n секунд, макс 3 попытки)
- [ ] Dead letter queue для превысивших лимит?
- [ ] Kill switch из пользовательского интерфейса?
- [ ] Identity checkpoint перед compaction? (self-model срез + post-compaction верификация)

## Проекция — не только cost

Failure case может быть про:
- **Identity drift** (агент потерял «кто я») → нужен identity checkpoint
- **Rogue loop без kill switch** → нужен watchdog/timer
- **Отсутствие oversight** (агент разослал 500 сообщений) → нужен confirmation gate
- **Credential leak** (75.8% агентов сливают ключи в stdout) → нужен secret scanner на выводе

## Питфоллы

- **Не лезь в код пользователя.** Твоя задача — показать failure mode и спроецировать. Код правит пользователь сам.
- **Не обещай, что сценарий случится.** Это вероятностная оценка, не предсказание.
- **Цифры проекции — с меткой «без учёта N».** Уточняй допущения: кэш-хит, средний размер запроса, частота ретраев.
- **Если пользователь не хочет разбирать — не настаивай.** Проекция нужна когда её просят, не когда тебе кажется важным.
