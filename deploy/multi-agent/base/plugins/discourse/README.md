# Discourse Plugin

Плагин для Hermes Agent — доступ к форуму на Discourse (по умолчанию —
support.wirenboard.com) **только для чтения**.

## Возможности

| Инструмент | Описание |
|------------|----------|
| `discourse_search` | Гибкий поиск: свободный текст + фильтры (категория, теги, автор, статус solved/open/closed, диапазон дат) |
| `discourse_get_topic` | Чтение целого треда: посты по порядку, автор, признак staff, текст (HTML → plain text), вложения/картинки на пост. Пагинация длинных тредов — автоматическая |
| `discourse_list_latest` | Лента последних активных тем, опционально по категории |
| `discourse_list_categories` | Список категорий (id/slug/название/read_restricted) — чтобы находить значения для фильтра `category` |
| `discourse_get_attachments` | Достаёт картинки/вложения из треда; с `download=true` — скачивает их локально и возвращает пути к файлам |

**Ответить, лайкнуть, отредактировать или удалить через этот плагин нельзя** —
ни одного write-тула здесь физически нет. Это осознанное решение: контент с
форума — untrusted (в тред может что угодно написать любой посетитель), и
если бы у бота был тул для ответа, пост на форуме теоретически мог бы
попытаться убедить модель что-нибудь написать (prompt injection). Раз тула
нет — этот путь атаки закрыт независимо от того, что бот "подумает".

## Установка

```bash
cp -r plugins/discourse ~/.hermes/plugins/discourse
hermes plugins enable discourse
# перезапусти Hermes
```

Работает и без токена — анонимные вызовы видят то же, что видит
незалогиненный посетитель форума (ровно то, что раньше использовал навык
`wb-support-portal-audit`). `DISCOURSE_API_KEY`/`DISCOURSE_API_USERNAME`
нужны, только если требуется доступ к закрытым категориям.

## Токен: как завести read-only доступ (по шагам)

Discourse требует пару `Api-Key` + `Api-Username`, привязанную к
конкретному пользователю форума. Поэтому сначала — пользователь, потом — ключ.

### 1. Завести служебного пользователя

Заходишь под своим админским аккаунтом на support.wirenboard.com.

- Открой **Admin → Users** (`/admin/users/list/active`).
- Если есть кнопка **"+ New"** (в современных версиях Discourse она есть
  специально для таких сервисных ботов) — жми её и создай пользователя
  напрямую: username (например `hermes-bot`), email, пароль. Админ может
  сразу активировать аккаунт, письмо-подтверждение не обязательно.
- Если кнопки нет — заведи аккаунт обычной регистрацией на отдельный email
  (например `hermes-bot@<ваш-домен>`), затем в Admin → Users найди его и
  при необходимости подтверди/активируй вручную (вкладка "Pending" если
  включена модерация новых юзеров).

**Важно после создания** — на странице пользователя (`/admin/users/<id>/<username>`):
- Trust Level оставь как есть (TL0/TL1, "basic user") — **не** делай его
  staff, moderator или admin.
- Не добавляй его ни в одну staff-группу.
- Если нужен доступ к конкретной закрытой категории — добавь его **только**
  в ту группу, у которой в настройках категории (Categories → Security)
  есть право See/Reply на эту категорию. Ничего сверх этого.

Смысл: даже если скоуп ключа где-то настроится шире, чем задумано, сам
аккаунт физически не сможет модерировать/удалять/отвечать от имени staff —
это вторая линия защиты, независимая от скоупа ключа.

### 2. Создать API-ключ со скоупом "Read Only"

- **Admin → API → Keys → New API Key**.
- **User Level**: выбери **"Single user"** и укажи того самого
  `hermes-bot` — не "All Users"/global, чтобы ключ не мог действовать от
  имени кого-то другого.
- **Scope**: выбери пресет **"Read Only"** (это готовый скоуп в Discourse,
  который разрешает только read-эндпоинты — просмотр тем, поиск, списки —
  и физически не даёт вызвать создание/редактирование поста). Не выбирай
  "Global".
- Сохрани, скопируй ключ сразу — Discourse показывает его один раз.

### 3. Проверить, что запись действительно заблокирована

```bash
KEY=... ; USER=hermes-bot

# должно вернуть JSON (200)
curl -s -H "Api-Key: $KEY" -H "Api-Username: $USER" \
  https://support.wirenboard.com/latest.json | head -c 200

# должно вернуть 403 — это и есть гарантия "read only", а не то,
# умеет ли бот вызывать POST
curl -s -o /dev/null -w "%{http_code}\n" -X POST \
  -H "Api-Key: $KEY" -H "Api-Username: $USER" \
  https://support.wirenboard.com/posts.json \
  -d "topic_id=1&raw=test"
```

Если второй вызов вернул `403` — всё настроено правильно.

### 4. Положить в `.env` инстанса

```
DISCOURSE_URL=https://support.wirenboard.com
DISCOURSE_API_KEY=...
DISCOURSE_API_USERNAME=hermes-bot
```

## Контракт

- Все вызовы — `GET` через `requests`, никакого `POST`/`PUT`/`DELETE` эндпоинта нигде в коде плагина нет.
- Без `DISCOURSE_API_KEY`/`DISCOURSE_API_USERNAME` — анонимные запросы (публичный контент, как раньше у `wb-support-portal-audit`).
- `discourse_get_attachments(download=true)` скачивает файлы в `<hermes_home>/data/discourse_downloads/topic_<id>/`, с капом ~20 МБ на файл.
- Токен читается из env, в коде не хранится.

## Примеры использования

> «Найди на форуме треды про modbus timeout за последний месяц, где ещё нет ответа»

> «Прочитай тред 12345 целиком и скажи, что просил пользователь»

> «Достань скриншоты из треда 12345 и посмотри на них»

> «Какие категории есть на форуме и в каких закрыт доступ?»
